1. xterm.js 支持哪几种渲染器？

xterm.js 历史上曾存在三种渲染器，截至 v6.0（2025-12-22 发布），仅保留两种：

渲染器类型状态 (v6.0+)说明
DOM Renderer 内置 活跃 每个终端单元格一个 <span> 节点，无需 GPU，始终可用。自 v5.0 起成为默认渲染器
Canvas Renderer addon → 已移除 v6.0 删除 v5.0 从核心移入 @xterm/addon-canvas；v6.0 通过 PR #5105 彻底移除
WebGL Renderer addon 活跃 @xterm/addon-webgl，使用 WebGL2 实现 GPU 加速。官方推荐的性能路径
演进路线：DOM（v1–v4 默认）→ Canvas 出现（PR #938, 2017）→ WebGL 出现（2019）→ Canvas 移出核心（v5.0, 2022）→ Canvas 彻底删除（v6.0, 2025）。现在的 xterm.js 只有两条渲染路径：DOM（回退）和 WebGL（性能）。
注意：v6.0 同时删除了 rendererType 选项和 setOption() 方法。此前用 rendererType: 'canvas' 的配置在 v6.0 中将直接报错。

2. 每种渲染器的工作原理和性能特点

2.1 DOM Renderer

工作原理：

  • 为屏幕上每个字符分配一个独立的 <span> 元素。一个 220 列 x 50 行的终端视口约产生 11,000+ 个 span 节点
  • 每帧 diff 当前与目标状态，仅更新变化的单元格
  • 字符属性（前景色、背景色、粗体、斜体、下划线等）通过 setAttribute 直接写入 DOM

性能瓶颈：

  • getAttribute / setAttribute 占据约 50% 脚本时间（约 800ms），详见 DOM renderer care package issue #4658
  • 大量 DOM 节点导致浏览器布局引擎反复重排（reflow），高吞吐输出时帧率降至 20-30 FPS
  • colortest 场景下渲染耗时约 3,000ms

2.2 Canvas Renderer（已移除）

工作原理：

  • 多个 <canvas> 元素按层叠放：Text（文字）→ Selection（选区）→ Link（链接下划线）→ Cursor（光标）
  • ASCII 字符预先渲染到 ImageBitmap 纹理图集（texture atlas），绘制时直接从图集采样，避免重复 fillText
  • 基于脏区域 diff 的增量渲染：仅重绘变化的单元格区域

性能数据（来源：PR #938，Chrome 60 / MacBook Pro）：

场景DOM (v2.9.2)Canvas提升倍数
键入单个字符4–5 ms0.4–0.5 ms~10x
vtop 空闲22–73 ms0.8–1.6 ms27–46x
ls -lR 大目录17–25 ms2.3–4.6 ms5–7x
yes（洪水输出）17–30 ms1–1.3 ms17–23x

单个字符变更通常 < 0.25 ms vs DOM 的 ~6ms。但 Canvas 仍是 CPU 侧 2D 渲染，WebGL 出现后沦为过渡方案。

2.3 WebGL Renderer

工作原理：

  • 单一 <canvas> 元素，使用 WebGL2 API
  • 整个终端视口的字符数据组织为 TypedArray，单次 drawElementsInstanced 调用全部推送到 GPU
  • CPU 耗时 >90% 用于数据解析/准备，<10% 用于实际渲染指令
  • 稳定维持 58-60 FPS

与其他渲染器的性能对比：

指标DOMCanvas (已移除)WebGL
帧率（高负载）20–30 FPS40–60 FPS58–60 FPS
colortest~3,000 ms—~50 ms (60x)
ls -lR17–25 ms2.3–4.6 ms<1 ms
vs Canvas 相对性能—基准最多 900% 更快

3. WebGL Renderer 如何利用 GPU 加速？

WebGL 渲染器通过四个核心技术实现 GPU 加速：纹理图集、实例化渲染、紧凑数据布局、分层渲染管线。

3.1 纹理图集 (Texture Atlas)

字符字形不逐次 fillText，而是预先光栅化到 GPU 纹理中：

  • 多页设计：每页从 512x512 起始，按需扩展至最大 4096x4096（v5.1.0 引入）
  • 页数上限：Math.min(32, gl.getParameter(gl.MAX_TEXTURE_IMAGE_UNITS))，硬上限 32 页，防止着色器代码过长导致编译失败（Issue #4273）
  • 纹理格式：RGBA / UNSIGNED_BYTE，CLAMP_TO_EDGE 包裹模式
  • 版本追踪：每页有版本号，仅在页内容变化时重新上传 texImage2D + generateMipmap
  • 共享缓存：CharAtlasCache 跨 WebGL 实例共享，引用计数管理生命周期

3.2 实例化渲染 (Instanced Rendering)

这是 WebGL 渲染器性能碾压 DOM/Canvas 的最关键设计：

  • 使用 drawElementsInstanced()，一次 draw call 渲染终端全部字符
  • 每个字符是一个 实例（instance），共享同一个模板四边形（unit quad）
  • 绘制基元为 TRIANGLE_STRIP（两个三角形拼成矩形）
  • 缓冲区提示为 STREAM_DRAW，适合每帧更新的场景

3.3 实例数据布局（11 floats / cell）

每个单元格携带 11 个 float（44 字节）的属性数据：

属性名类型含义
a_offsetvec2字形在单元格内的偏移量
a_sizevec2字形宽高
a_texpagefloat纹理图集页码索引
a_texcoordvec2纹理坐标 (UV)
a_texsizevec2纹理空间中的尺寸
a_cellposvec2单元格行列位置

这些属性以交叉存储（interleaved）方式组织在单个缓冲区中，vertexAttribDivisor 设为 1 实现逐实例而非逐顶点的数据读取。

3.4 着色器管线

顶点着色器（GLSL 300 es）：

  • 将单元格坐标从像素空间映射到裁剪空间（u_projection * vec4(zeroToOne, 0, 1)）
  • 输出 v_texcoord（UV坐标）和 v_texpage（flat int，纹理页号）

片段着色器（动态生成）：

  • 根据 maxAtlasPages 运行时生成 if/else 分支链
  • 接收 sampler2D u_texture[N] 数组（N = 纹理页数）
  • 根据 v_texpage 选择正确的采样器，执行 texture(u_texture[i], v_texcoord)
  • 精度限定符：precision lowp float

矩形着色器（背景色/选区）：

  • 独立着色器程序，接收 a_position、a_size、a_color
  • 仅输出纯色，无纹理采样

3.5 分层渲染 + 双缓冲

渲染顺序（由 WebglRenderer 协调）：

RectangleRenderer (背景色、选区) → GlyphRenderer (文字字形) → LinkRenderer (链接下划线) → CursorRenderer (光标，v5.3.0 起移入 GPU 画布)

使用 双缓冲机制，STREAM_DRAW 提示让驱动优化"每帧覆盖"的使用模式。

4. WebGL addon 的架构设计

4.1 源码结构

所有 WebGL addon 源码位于：addons/addon-webgl/src/

addons/addon-webgl/src/ ├── WebglAddon.ts # 入口：激活/停用/dispose ├── WebglRenderer.ts # 总协调器：管理渲染层、纹理图集、帧循环 ├── GlyphRenderer.ts # 字形渲染：编译着色器、管理实例顶点缓冲区 ├── RectangleRenderer.ts # 矩形渲染：背景色、选区高亮 ├── LinkRenderer.ts # 链接下划线渲染 ├── CursorRenderer.ts # 光标渲染 (v5.3.0 起原生 GPU 渲染) ├── RenderModel.ts # 单元格模型：Uint32Array 存储每cell状态 ├── WebglCharAtlas.ts # 纹理图集：多页纹理管理、版本追踪 ├── WebglUtils.ts # 工具：createProgram、投影矩阵等 └── atlas/ └── CharAtlasCache.ts # 图集缓存：跨实例共享、引用计数

4.2 核心类关系

WebglAddon
  └── activate(terminal) → 创建 WebglRenderer
                              ├── GlyphRenderer      (主要渲染)
                              │     ├── 动态生成片段着色器
                              │     ├── 11-float 实例化顶点缓冲区
                              │     └── 纹理引用 → WebglCharAtlas
                              ├── RectangleRenderer  (背景层)
                              ├── LinkRenderer       (链接层)
                              ├── CursorRenderer     (光标层)
                              ├── RenderModel        (Uint32Array 状态存储)
                              └── WebglCharAtlas     (多页纹理图集)
                                    └── 注册到 CharAtlasCache

4.3 RenderModel -- 单元格状态存储

RenderModel 使用 Uint32Array 存储每个单元格的渲染状态，每个单元格 4 个 uint32：

索引字段含义
0bgColor背景颜色 (packed RGBA)
1fgColor前景颜色 (packed RGBA)
2charCode字符码点 + 合并字符标记位 (COMBINED_CHAR_BIT_MASK = 0x80000000)
3extAttrs扩展属性 (粗体/斜体/下划线/闪烁/反显等)

总大小：rows * cols * 4 * 4 字节。220x50 终端约 176 KB。

4.4 纹理图集缓存

CharAtlasCache 是关键共享基础设施：

  • 跨 WebglAddon 实例共享：同一应用内多个终端可复用缓存
  • 引用计数：acquireTextureAtlas() 获取，releaseTextureAtlas() 释放
  • 多页管理：WebglCharAtlas 内部维护页面数组，懒加载新页
  • 版本增量更新：页面 canvas 变更时递增版本号，渲染器检测版本不匹配后上传纹理

4.5 着色器编译策略

片段着色器在 GlyphRenderer 构造函数中动态生成：

  • 从 gl.MAX_TEXTURE_IMAGE_UNITS 获取最大纹理单元数
  • 硬上限 32 页（防止某些平台 128 单元导致的"Expression too complex"编译错误）
  • 生成 if/else 链，每个分支绑定一个纹理单元：if (v_texpage == N) { outColor = texture(u_texture[N], v_texcoord); }
  • 编译失败时捕获错误并回退到 DOM

5. 官方推荐选择

5.1 基本原则：WebGL 优先，DOM 兜底

推荐模式（来自 @xterm/addon-webgl 官方文档）：
  1. 尝试加载 WebglAddon
  2. 加载失败 → 静默回退到 DOM 渲染器（无需任何操作，DOM 为内置默认）
  3. 运行时 GPU 上下文丢失 → onContextLoss 触发 dispose()，自动回退 DOM

5.2 标准接入代码

import { Terminal } from '@xterm/xterm';
import { WebglAddon } from '@xterm/addon-webgl';

const term = new Terminal();
term.open(document.getElementById('terminal'));

try {
  const webglAddon = new WebglAddon();
  webglAddon.onContextLoss(e => {
    webglAddon.dispose();  // 终端自动回退到 DOM 渲染器
  });
  term.loadAddon(webglAddon);
} catch (e) {
  // WebGL2 不可用，终端默认使用 DOM 渲染器，无需额外处理
  console.warn('WebGL2 不可用，使用 DOM 渲染器');
}

5.3 GPU 硬件检测

在加载 WebglAddon 之前，建议先检测真实 GPU 硬件是否存在：

function detectWebGL2() {
  const canvas = document.createElement('canvas');
  const gl = canvas.getContext('webgl2', {
    failIfMajorPerformanceCaveat: true  // 关键：拒绝软件渲染器
  });
  return !!gl;
}

Chrome/Chromium 存在一个有问题的软件 WebGL2 模拟器（Issue #4574），会导致字符缺失或渲染异常。failIfMajorPerformanceCaveat: true 确保只有真实 GPU 才走 WebGL 路径。

5.4 上下文丢失处理

事件处理策略
onContextLoss调用 addon.dispose()，终端自动回退 DOM。无需手动切换
loadAddon 抛异常捕获、记录日志、继续使用 DOM。用户无感知
GPU 进程崩溃自动触发 onContextLoss，同上处理
系统休眠/唤醒浏览器可能回收上下文，触发 onContextLoss

5.5 多面板终端策略

Chromium 每个渲染器进程限制约 16 个活跃 WebGL 上下文。对于多面板终端应用（如 VS Code、终端模拟器），推荐使用 单上下文聚焦模型（来自 daintree#3156）：

  • 仅聚焦终端使用 WebGL 渲染
  • 所有非聚焦终端使用 DOM 渲染器
  • 焦点切换时：dispose 旧 WebGL 上下文 → 在新终端创建新 WebglAddon
  • 加入去抖动（50-500ms）防止快速切换引起频繁重构
  • 稳定状态：整个应用中仅有 1 个活跃 WebGL 上下文
// 多面板下的焦点切换策略（概念代码）
let activeWebglAddon: WebglAddon | null = null;

function onTerminalFocus(gained: Terminal, lost: Terminal) {
  // 先释放旧上下文
  activeWebglAddon?.dispose();
  activeWebglAddon = null;

  // 为新聚焦终端创建 WebGL
  try {
    const addon = new WebglAddon();
    addon.onContextLoss(() => addon.dispose());
    gained.loadAddon(addon);
    activeWebglAddon = addon;
  } catch {
    // 静默回退 DOM
  }
}

7. xterm.js 5.x/6.x 渲染方面的重大变化

7.1 v5.0.0（2022-11-01）

来源：Release 5.0.0

  • Canvas 渲染器从核心移至 addon：从 xterm 核心包中剥离，变为独立包 xterm-addon-canvas（后改为 @xterm/addon-canvas）
  • 核心包体积缩减：379 KB → 265 KB（减少约 30%）
  • DOM 成为新默认渲染器：原默认是 Canvas，现改为 DOM
  • 新下划线样式：支持 undercurl、dotted、dashed、double 等
  • Smooth scroll：平滑滚动（使用 transform/scrollTop）
  • Powerline 字形改进：支持更多 Powerline 特殊字符
  • 装饰（Decorations）系统：浮层式 UI 元素，不污染终端内容层

7.2 v5.1.0（2022-12-19）

来源：Release 5.1.0、Discussion #4334

  • 多页纹理图集：从单页 1024x1024 升级为多页系统，每页 512x512 起始、最大 4096x4096。理论上无限字形空间
  • Canvas / WebGL 共享纹理图集：统一图集代码，降低重复维护成本
  • WebGL 在 Safari 16+ 启用：此前 Safari 不支持 WebGL2，是 Canvas 渲染器保留的主要原因之一
  • 延迟纹理图集预热：使用 requestIdleCallback 推迟启动时的图集构建，加快首屏
  • 8 个新 Powerline Extra 字形

7.3 v5.3.0（2023-09-08）

来源：Release 5.3.0

  • DOM 渲染器大幅加速：多个 PR（#4605, #4631, #4651, #4662, #4681, #4754）修复了 setAttribute 瓶颈、改进选择渲染一致性等
  • WebGL 光标迁移到 GPU 画布：光标不再使用独立 DOM 覆盖层，直接在 WebGL 画布中渲染（PR #4568），提升 GPU 模式下的光标渲染性能和一致性
  • 纹理图集利用率优化：修复字形合并时的损坏问题
  • smooth scroll 扩展到 scrollLines、scrollPages 等方法

7.4 v5.4.0（2023-12）

来源：Release 5.4.0

  • 包名迁移至 @xterm 作用域：xterm → @xterm/xterm，xterm-addon-webgl → @xterm/addon-webgl，xterm-addon-fit → @xterm/addon-fit 等。旧包名全部标记为 deprecated
  • 迁移原因：安全——作用域包消除命名歧义，防止 typosquatting 攻击
  • 对渲染无直接影响，但所有后续版本均使用新包名

7.5 v6.0.0（2025-12-22）-- 渲染器重大分水岭

来源：Issue #4779、PR #5105、VS Code #106202

v6.0.0 是 xterm.js 渲染架构的决定性版本。Canvas 渲染器被彻底删除，渲染路径从三条缩减为两条。

详细变更：

  • @xterm/addon-canvas 完全移除（PR #5105）：团队称"多年前就想删除"，Safari WebGL2 支持是最后的绊脚石。移除约 1-2k 行代码，减少维护负担
  • rendererType 选项删除：不再可配置渲染器类型。DOM 始终是内置回退，WebGL 通过 addon 启用
  • setOption() 方法删除：改为直接 options 属性赋值
  • windowsMode 选项删除：不再可用
  • bellStyle 选项删除：替换为 onBell 事件回调
  • Viewport/scrollbar 重构（PR #5096）：行为和 API 发生变化

迁移影响：

  • VS Code 将设置项从 terminal.integrated.rendererType（选项：auto/canvas/dom/experimentalWebgl）改为 terminal.integrated.gpuAcceleration（选项：auto/on/off）
  • RStudio 等大量下游项目在升级时主动接入 WebGL addon（rstudio#16820）
  • 如果旧代码引用了 rendererType: 'canvas'，升级到 v6.0 会直接报错
