1. CI 验证（GitHub Actions）

1.1 验证目标

确认 GitHub Actions 的 windows-latest CI pipeline 全绿通过，覆盖：

  • Lint 检查（eslint src/）
  • 前端构建（npm run build，Vite 产物）
  • 前端测试 L2（Vitest 单元/组件测试）+ L3（@xterm/headless 渲染测试）
  • 后端构建（cargo build）
  • 后端测试 L1（cargo test）
  • Tauri debug 构建（tauri build --debug --no-bundle）
  • E2E 测试 L4（WebDriver + msedgedriver，continue-on-error: true）
注意

/goal 评判器看不到 GitHub 页面——此项需人工确认。完成后将 CI run URL 贴入终端供引用即可。

CI 常见失败排查
  • ESLint 报错：检查是否有未使用的 import 或类型错误；运行 npx eslint src/ --fix 自动修复
  • cargo build 失败：确认 Cargo.toml 依赖版本与 plan/version-pins.md 一致；本地先 cargo check
  • Tauri build 失败：检查 tauri.conf.json 的 identifier、frontendDist 路径是否正确
  • msedgedriver 安装失败：chippers/msedgedriver-tool 为社区工具，若 CI 上版本检测异常可暂时跳过（E2E 有 continue-on-error）

2. L4 E2E 测试（条件性）

2.1 验证目标

通过 Tauri WebDriver 协议 + msedgedriver 执行端到端测试，验证应用启动、窗口创建、前端渲染等完整链路。

关键限制

Playwright 不可用于 Tauri——Tauri 使用系统 WebView2（非 Chromium），Playwright 无法驱动。只能走 tauri-driver + WebdriverIO + msedgedriver（WebDriver 协议）。

2.2 前提条件

条件检查命令不满足时的操作
Rust 工具链已安装 rustc --version rustup install stable
tauri-driver 已安装 tauri-drriver --version 见下方安装步骤
msedgedriver 已安装且版本对齐 WebView2 msedgedriver-tool.exe 见下方安装步骤
应用 debug build 产物存在 检查 src-tauri\target\debug\slterminal.exe npm run tauri build -- --debug --no-bundle

2.3 操作步骤

1 安装 tauri-driver

cargo install tauri-driver --locked

版本参考：tauri-driver 2.0.6（见 plan/version-pins.md）

钉表红旗 #7

tauri-driver 不自动追加 .exe 后缀（已知 issue #11317）。wdio.conf.js 已使用完整路径 path.resolve(os.homedir(), '.cargo', 'bin', 'tauri-driver') 规避。host 使用 127.0.0.1 而非 localhost。

2 安装 msedgedriver

# 安装社区版 msedgedriver 检测/下载工具
cargo install --git https://github.com/chippers/msedgedriver-tool

# 运行工具——自动检测 WebView2 版本并下载对应 msedgedriver
msedgedriver-tool.exe

3 确认 WebView2 与 msedgedriver 版本对齐

关键版本基线（来自 plan/version-pins.md 红旗清单）：

组件最低版本项目基线
WebView2 Runtime101.0.1210.39110.0.1531.0
msedgedriver（与 WebView2 匹配）同 WebView2 版本号
# 查看本机 WebView2 版本
Get-ItemProperty "HKLM:\SOFTWARE\WOW6432Node\Microsoft\EdgeUpdate\Clients\{F3017226-FE2A-4295-8BDF-00C3A9A7E4C5}" | Select-Object pv

# 查看 msedgedriver 版本
msedgedriver --version
红旗 #7：版本检测依赖社区工具

chippers/msedgedriver-tool 为社区维护，非 Tauri 官方。如果版本检测异常，可手动从 Microsoft Edge WebDriver 页面 下载对应版本的 msedgedriver.exe 并放入 PATH。

4 构建应用（debug 模式）

npx wdio run wdio.conf.js

3. 人工验证

3.1 验证目标

启动 Tauri 开发模式，肉眼确认应用窗口的基本呈现：暗色主题、Dockview 空白布局、无控制台报错。

3.2 前提条件

条件状态
Tauri debug build 已完成产物在 src-tauri\target\debug\slterminal.exe
前端依赖已安装node_modules/ 存在（npm ci 已执行）
Rust 后端可编译cargo check 通过（在 src-tauri/ 目录）

3.3 操作步骤

1 启动应用（开发模式）

# 在项目根目录执行
npm run tauri dev

Tauri dev 会同时启动：

  • Vite dev server（前端热更新，默认 localhost:1420）
  • Rust 后端（编译 + 启动系统 WebView2 窗口）

2 肉眼确认清单

#检查项通过标准
A 窗口以暗色主题呈现 窗口背景色为暗色（非白色/亮色），标题栏、滚动条等 chrome 元素也跟随暗色
B Dockview 区域填满窗口 窗口客户区无空白/白边，Dockview 布局容器填充整个可用区域；布局为空（无面板标签页），显示暗色空状态
C 控制台无报错 启动 Tauri dev 的终端输出中无 ERROR、panic、unreachable 等异常；WebView2 DevTools 控制台无红色错误（按 F12 或 Ctrl+Shift+I 打开）

3.4 预期结果

通过标准

暗色空 Dockview 窗口，无报错。

一个以 JetBrains 暗色调为背景的空布局窗口，Dockview 容器正确填充客户区，前后端通信正常（IPC 链路无错误），WebView2 渲染无异常。

3.5 异常排查

异常现象可能原因排查动作
窗口为白色/亮色 暗色主题未正确注入 检查 index.html 的 <body> 是否设置 background；检查 CSS 变量是否正确加载
窗口内白屏/空白 Vite dev server 未就绪 或 frontendDist 指向错误 确认 Vite 端口 1420 未被占用；检查 tauri.conf.json 的 devUrl 指向正确
Dockview 未填充窗口 CSS 布局问题 确认根容器使用 width: 100vw; height: 100vh 或 flex 填充
Rust panic 后端初始化错误（如 WebView2 不可用） 确认系统已安装 WebView2 Runtime（Win11 预装）；查看 panic 信息定位模块
WebView2 控制台报错 前端 JS 异常 按 F12 打开 DevTools，查看 Console 标签页的具体错误信息
附：Tauri 窗口初始配置参考（tauri.conf.json） 
{
  "app": {
    "windows": [
      {
        "title": "slTerminal",
        "width": 1200,
        "height": 800,
        "resizable": true,
        "fullscreen": false,
        "theme": "dark"
      }
    ]
  }
}

确认 "theme": "dark" 已设置——这会让窗口标题栏跟随系统暗色主题。
