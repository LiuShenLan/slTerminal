# Claude Code 终端 — 技术架构设计文档

> 方案:Scheme B(Tauri 2 + Rust 核心 + Web UI)。本文给出整体架构、后端/前端分层、关键数据流、跨切面设计,以及**模块文件路径划分与开发约束**(用于限制后续开发)。
> 参考成熟案例:Terax(Tauri 2 + portable-pty + React + xterm.js + CodeMirror 6,与本项目目标几乎一致)、Tauri 官方项目结构与最佳实践、VS Code 多进程模型、Dockview。
> 配套文档: [1. 需求与分阶段计划](./1. 需求与分阶段计划.md)、[2. 详细方案与最小demo.md](./2. 详细方案与最小demo.md)

-----

## 一、架构总览

两进程模型:**Rust 后端拥有一切 OS 访问;Web 前端只做 UI,经 IPC 调用后端。**

```
┌──────────────────────────────────────────────┐
│ 前端 (WebView2, React + TS)                    │
│  ┌─ 主侧栏(项目→工作区/worktree→操作页面)     │
│  ┌─ 次级侧栏(文件浏览器, git 着色)             │
│  ┌─ Dockview 工作区(操作页面 = 一套布局)       │
│  │     面板 = Terminal(xterm.js) / Editor(CM6) │
│  └─ 底部状态栏(分支/diff/python)              │
│            │ 只经 src/ipc/* 通信               │
└────────────┼─────────────────────────────────┘
   invoke()  │  ▲ Channel<Event> / emit
┌────────────┼─────────────────────────────────┐
│ 后端 (Rust, src-tauri)                          │
│  AppState ── pty / fs / git / claude / notify   │
│  portable-pty(ConPTY) · git2 · notify · 系统通知 │
└────────────────────────────────────────────────┘
```

分层原则:

- **唯一通信通道**:前端任何 OS 能力都只经 `src/ipc/*` → 后端命令;UI 层不碰系统 API。
- **后端按功能分模块**:每个领域(pty/fs/git/claude/notify)自带命令、状态、DTO;模块间不互相穿透,共享只经 `AppState`。
- **状态单一来源**:终端会话、项目树、布局各有唯一 store(前端)/State(后端);组件只订阅,不持有真值。

## 三、后端架构(Rust)

### 模块与职责

- **pty**:终端会话生命周期。命令 `pty_spawn/write/resize/kill`;状态 `PtyState = RwLock<HashMap<SessionId, Session>>`;输出经 `Channel<PtyEvent>` 推送;spawn 时注入 PowerShell profile(发 OSC 7 / 133 + venv 序列)与唯一 `SLTERM_PANE_ID` 环境变量(供 claude hook 关联)。
- **fs**:文件浏览器 + 编辑器 IO(读目录/读写文件/stat/rename/delete);`notify` 文件监听发事件。
- **git**:`git_status / git_diff / git_branch`(用 git2);worktree `add/list/remove` **走 shell `git worktree`**(git2 对建新分支支持弱);自建 worktree 落 `.claude/worktrees/<name>`、分支 `worktree-<name>`(对齐 claude `--worktree`,使隐藏逻辑与 `WorktreeCreate` 检测一致),并确保 `.claude/worktrees/` 入 `.gitignore`。
- **claude**:会话↔页签映射(经 PTY 注入的 `SLTERM_PANE_ID` 由 hook 回传精确归属);`hooks`(向全局 `~/.claude/settings.json` 装 `type:http` 钩子,事件 POST 到本应用本地 HTTP 服务、`async` 不阻塞 claude;事件 → 活动状态用锁存状态机);`transcript`(只读 `~/.claude/projects/.../*.jsonl`);`statusline`(仅提供脚本,渲染在 claude TUI)。
- **notify**:桌面 toast(需介入 / 完成)。
- **app/state/error**:`AppState` 聚合各模块状态;`AppError`(thiserror + Serialize)。

### 命令与事件

- 命令:`#[tauri::command] async fn` → 统一在 `lib.rs` 的 `generate_handler!` 注册;一律返回 `Result<_, AppError>`。
- 事件:优先用 **Tauri `Channel<T>`**(有序、定向单会话、开销小)做高频流(PTY 输出、hooks 状态);低频用全局 `emit`。
- 参数命名:Rust `snake_case` ↔ JS `camelCase`(Tauri 自动转换)。

### 并发与线程

- 每个 PTY 一个读线程,字节流 → Channel;**spawn 用 `SPAWN_LOCK` 串行化**(并发 spawn 会卡 ConPTY 输出管道)。
- 阻塞 I/O(文件、git)用 `spawn_blocking`,不阻塞 async 运行时。

### 安全

- `capabilities/` 最小权限;每新增命令需显式放行。CSP 严格。所有来自前端的输入在 Rust 侧校验。
- **claude hook 回传服务**:本地 HTTP 仅绑 `127.0.0.1` + 随机端口;每次启动生成随机 token,hook 经 `Authorization` header 校验,拒绝无 token 请求(防本机其它进程伪造事件)。hook 装入全局 `~/.claude/settings.json`,内部以 `SLTERM_PANE_ID` 是否存在为门(模拟器外运行的 claude 即 no-op),安装可逆。

## 四、前端架构(Web)

### 分层

1. **ipc/**:唯一与后端通信处。`invoke` 只在此出现,对外暴露领域函数(`pty.spawn()` 等)。
1. **types/**:与 Rust DTO 一一对应的 TS 类型。
1. **stores/**(Zustand):`sessions`(会话↔面板、活动状态)、`projects`(项目/工作区/worktree 树)、`layout`(操作页面↔Dockview 布局)、`settings`。不放 UI。
1. **workspace/**:Dockview 容器 + 面板注册表 + 布局存取(`toJSON/fromJSON`)。
1. **panels/**:每种 Dockview 面板 = 一个组件(`terminal/`、`editor/`)。终端的 xterm 生命周期、编辑器的 CM6 + diff 边栏都封装在各自目录。
1. **features/**:非面板的 UI(主侧栏、文件浏览器、状态栏、worktree 管理、通知、历史会话)。
1. **theme/**:暗色 token;**JetBrains 两套 git 配色集中在 `theme/colors.ts`**。
1. **lib/**:纯工具(无副作用、无 IPC)。

### 工作区模型

- 一个**操作页面** = 一份 Dockview 布局 JSON;切换操作页面 = `fromJSON` 换布局。
- **面板只能是 `panels/` 里注册过的类型**;终端与文件编辑器是同一套页签系统里的对等面板(拖拽分屏/嵌套/调比例由 Dockview 提供)。
- 每个 `xterm.Terminal` 随面板存活,切走只隐藏 canvas、不销毁会话。

## 五、关键数据流

**终端会话**:面板 → `ipc.pty.spawn({shell,cwd,cols,rows})` → 后端 spawn + 存 `PtyState` + 开 Channel → 读循环 `Output{bytes}` → `term.write` → `term.onData` → `ipc.pty.write` → PTY;resize 经 fit → `pty_resize`;退出 → `Exit{code}` → 标记面板 + 更新 `sessions` store。

**打开文件**:浏览器双击 → `ipc.fs.readFile` → CM6 面板;保存 → `ipc.fs.writeFile`;claude 改了已打开文件(`notify` 监听)→ 干净则重载、脏则弹窗选择。

**git/python 状态**:`notify` 监听仓库变化 → `ipc.git.status/diff` → 文件名着色(浏览器)+ 底部状态栏(分支/diff)+ 编辑器行内 diff;python 环境感知经 PTY profile 发的**自定义 OSC 序列**回传当前激活 venv/conda(宿主读不到子 shell env),同样进底部状态栏。

**claude 活动状态**:claude 的 `type:http` hook 把事件 POST 到本地服务(header 带 `SLTERM_PANE_ID`)→ `claude::hooks` 按 `pane_id` 归属页签、用**锁存状态机**映射(`UserPromptSubmit`/`PreToolUse` → 运行中,锁存至 `Stop` → 完成;`PermissionRequest`/`Elicitation` → 需介入;`Notification(idle_prompt)` → 等输入;`SubagentStop` 不算完成)→ 更新 `sessions` store → 终端页签状态点 + 操作页面节点聚合(需介入 > 运行中 > 空闲);需介入/完成且对应页签未聚焦 → `notify` 桌面通知。

-----

## 六、跨切面

- **主题**:仅暗色;颜色 token 集中,组件不硬编码。
- **错误**:后端 `AppError` → 前端统一处理与提示;前端全局错误边界。
- **日志**:后端结构化日志(tracing);前端关键路径打点。
- **配置持久化**:跨重启持久化 **项目列表 / 各项目操作页面结构(L3 名)/ 用户设置 / 键位**,经后端写入用户配置目录(经 fs 命令)。**不**持久化页签内容与 PTY 会话:重启后操作页面回到空白"打开终端"页(结构仍在)。
- **性能**:PTY 事件合批后再渲染(大输出防卡);xterm WebGL,驱动异常可切 canvas 回退。
- **平台分支**:`#[cfg(windows)]` 只允许出现在 pty 的 spawn/shell 等明确处,业务逻辑不撒 cfg。

## 七、项目模块文件路径划分(开发约束)

> 这是对后续开发的硬性结构约束。新增功能必须落到既有目录与规则里,不另起炉灶。

```
claude-terminal/
├── package.json  index.html  vite.config.ts  tsconfig.json
├── src/                              # 前端
│   ├── main.tsx                      # 入口
│   ├── App.tsx                       # 根: 主侧栏 + 次级侧栏 + 工作区 + 底部状态栏
│   ├── ipc/                          # ★ 唯一与后端通信的层(invoke 只在此)
│   │   ├── client.ts                 #   invoke 封装、Channel 订阅基建
│   │   ├── pty.ts  fs.ts  git.ts  claude.ts
│   ├── types/                        # 与 Rust DTO 对应的 TS 类型
│   │   └── index.ts
│   ├── stores/                       # 状态(Zustand),不放 UI
│   │   ├── sessions.ts  projects.ts  layout.ts  settings.ts
│   ├── workspace/                    # Dockview 工作区外壳
│   │   ├── Workspace.tsx  panelRegistry.ts  layoutSerde.ts
│   ├── panels/                       # ★ 每种面板一个目录,须在 panelRegistry 注册
│   │   ├── terminal/  (TerminalPanel.tsx, useXterm.ts, theme.ts)
│   │   └── editor/    (EditorPanel.tsx, useCodeMirror.ts, gitGutter.ts)
│   ├── features/                     # 非面板 UI
│   │   ├── sidebar/  explorer/  statusbar/  worktree/  notifications/  history/
│   ├── theme/                        # ★ 暗色 token;JetBrains git 配色集中在此
│   │   ├── colors.ts  dark.css
│   └── lib/                          # 纯工具(无副作用、无 IPC)
└── src-tauri/                        # 后端
    ├── Cargo.toml  build.rs  tauri.conf.json
    ├── capabilities/default.json     # 最小权限
    ├── icons/
    └── src/
        ├── main.rs                   # 仅 app_lib::run(),不改
        ├── lib.rs                    # 注册命令/插件/State,run()
        ├── error.rs                  # AppError(thiserror + Serialize)
        ├── state.rs                  # AppState(聚合各模块状态)
        ├── pty/   (mod.rs 命令, session.rs, spawn.rs, io_loop.rs, shell.rs)
        ├── fs/    (mod.rs 命令, watch.rs)
        ├── git/   (mod.rs 命令, worktree.rs)
        ├── claude/(mod.rs, hooks.rs[装 hook + 本地 HTTP 回传服务], transcript.rs, statusline.rs)
        └── notify/(mod.rs)
```

### 约束清单(开发时必须遵守)

1. **前端绝不直接碰 OS/文件/进程**。`invoke` 只允许出现在 `src/ipc/`;其它任何文件只能调用 `ipc/` 暴露的领域函数。
1. **后端按功能分模块**(pty/fs/git/claude/notify)。每个模块自带命令、状态、DTO;模块间不互相 reach into,共享只经 `state.rs` 的 `AppState`。
1. **命令统一注册**于 `lib.rs` 的 `generate_handler!`;一律 `Result<_, AppError>`;阻塞 I/O 用 `spawn_blocking`。
1. **DTO 双边对应**:`src/types/` ↔ 各 Rust 模块的 DTO 一一对应;Rust `snake_case` ↔ JS `camelCase`,改一边必须改另一边。
1. **面板封闭**:Dockview 面板只能是 `panels/` 下注册过的类型;新增内容类型 = 加目录 + 在 `panelRegistry.ts` 注册,不在别处实例化。
1. **配色单点**:JetBrains 两套 git 色(文件名色、行内 diff 色)只在 `theme/colors.ts` 定义;组件引用 token,禁止硬编码颜色。
1. **布局单点**:操作页面布局只经 `workspace/layoutSerde.ts` 用 Dockview `toJSON/fromJSON` 存取,不手搓布局结构。
1. **会话单点**:会话真值(id/活动状态/cwd)只在 `stores/sessions.ts`;面板组件只订阅,不自存。
1. **平台分支收敛**:`#[cfg(windows)]` 只允许出现在 `pty/spawn.rs`、`pty/shell.rs` 等明确处;ConPTY spawn 串行化、cwd 反斜杠规范化都在 `pty/spawn.rs`。
1. **权限最小化**:新增命令必须在 `capabilities/` 显式放行;不使用通配 `*`。

## 八、与分阶段计划的对应(落地顺序)

- 最小 demo:`src-tauri/src/pty/*` + `src/ipc/pty.ts` + `src/panels/terminal/*` + `src/workspace/*`(Dockview 单面板)。
- 之后每个功能都落进既有目录:开文件→`panels/editor/` + `fs/`;侧栏/项目→`features/sidebar/` + `stores/projects.ts`;worktree→`git/worktree.rs` + `features/worktree/`;状态栏→`features/statusbar/` + `git/`;claude→`claude/*` + `features/notifications/`。

> 原则不变:**功能增长 = 往既定模块加文件、往 IPC 加命令、往 panelRegistry 加面板**,目录边界与上述约束始终保持。
