# Claude Code 终端 — 完整开发计划

> 综合[1. 需求与分阶段计划](./1. 需求与分阶段计划.md)、[2. 详细方案与最小demo.md](./2. 详细方案与最小demo.md)、[3. 技术架构设计文档.md](./3. 技术架构设计文档.md)。每个阶段给出:**开发内容 / 注意事项 / 验收**(验收区分**自动化**与**人工**,尽量自动化)。
> 总原则:能自动化就不靠人;人工只留给渲染保真、输入手感、配色肉眼、与真实 claude 联动这些机器够不到的部分,并尽量用录制样本 + 黄金快照把人工面缩到最小。

## 一、验收方法学(全程通用)

|层             |工具                                                          |覆盖                                                 |类型            |
|--------------|------------------------------------------------------------|---------------------------------------------------|--------------|
|**L1 后端单测/集成**|`cargo test`(+ Tauri mock runtime)                          |纯逻辑、PTY 往返、git/fs、状态、错误                            |自动            |
|**L2 前端单测/集成**|Vitest + `@tauri-apps/api/mocks`(`mockIPC`)                 |UI 逻辑、IPC 调用、store、面板接线(mock 后端,jsdom 里跑)          |自动            |
|**L3 终端缓冲区**  |`@xterm/headless` + `@xterm/addon-serialize`(准线;Termless 仍 0.x,作可选)|喂字节→断言渲染缓冲/单元格/光标/滚动/颜色;键位编码字节                     |自动            |
|**L4 端到端**    |`tauri-driver` + WebdriverIO(Windows: msedgedriver)         |启动真应用、驱动 DOM、断言关键流程                                |自动(较重,只覆盖关键路径)|
|**L5 日志/不变量** |`tracing` 结构化日志 + 运行时断言 + CI 日志检查                           |spawn 串行化、无孤儿 PTY、无未捕获错误等难单测项                      |半自动           |
|**M 人工**      |人                                                           |渲染保真、**中文 IME**、claude TUI 实测、配色肉眼比对、拖拽手感、通知外观、延迟手感|人工            |

补充:

- **CI**:GitHub Actions `windows-latest` 矩阵,每次 push/PR 跑 L1–L3(快),L4 跑关键流程;失败即拦截合并。
- **缩小人工面的手段**:L3 用录制的 claude 输出样本做黄金快照;配色用 token 断言(见架构约束第 6 条)把”肉眼比色”变成”断言 token”;E2E 可接 MCP 让 agent 驱动,进一步自动化。
- `mockIPC` 也能模拟事件/Channel 流(用回调发 `Output`/`Exit` 负载),所以终端接线在没有真 PTY 时也能自动验。

## 二、阶段计划

### Phase 0 — 工程与测试基建

**开发内容**

- Tauri 2 + React + TS + Vite 脚手架(暗色);按架构文档铺好目录骨架(`ipc/ stores/ workspace/ panels/ features/ theme/ lib/`;`src-tauri` 下 `pty/ fs/ git/ claude/ notify/ error.rs state.rs lib.rs`)。
- 测试基建一次性搭好:`cargo test`;Vitest + `@tauri-apps/api/mocks`;`@xterm/headless` + `@xterm/addon-serialize`(主用;Termless 0.x 可选);`tauri-driver` + WebdriverIO(配 msedgedriver);GitHub Actions CI(windows-latest)。
- `AppError`(thiserror + Serialize)、`capabilities/default.json` 骨架、`tracing` 日志。

**注意事项**

- 锁定工具链版本;CI 里让 msedgedriver 版本与 WebView2 同步。
- 立规矩(写进 lint/约定):`invoke` 只允许出现在 `src/ipc/`;命令一律 `Result<_, AppError>`。

**验收 — 自动化**

- `cargo test`、`npm test`(Vitest)可运行;各放一个样例:mockIPC 测试通过、`@xterm/headless` serialize 测试通过。
- `tauri build --debug` 成功;CI 在 windows-latest 全绿。

**验收 — 人工**

- 应用窗口能打开,呈现暗色空 Dockview。

### Phase 1 — 核心外壳(M1)

#### 1.1 PowerShell 终端面板

**开发内容**:`pty/`(`pty_spawn/write/resize/kill` + 读循环 → `Channel<PtyEvent>` + `PtyState`;`shell.rs` 注入 PowerShell profile 发 OSC 7/133 + venv 序列;spawn 注入唯一 `SLTERM_PANE_ID` 环境变量)、`ipc/pty.ts`、`panels/terminal/`(xterm + webgl + fit)、Dockview 单面板。
**注意事项**:`SPAWN_LOCK` 串行化 spawn;cwd 规范化为 `\`;shell 优先级 pwsh→powershell→cmd;对 PTY 事件**合批渲染**(按帧 rAF≈16ms 或字节阈值触发,减少 `term.write` 次数)防大输出卡死;每个 `xterm.Terminal` 随面板存活、切走只隐藏。
**验收 — 自动化**:

- L1:spawn pwsh → 写 `echo hello\n` → 断言输出流含 `hello`;`pty_resize` 改变行列;`pty_kill` 结束会话且**无孤儿进程**(L5 日志/进程断言)。
- L2:面板挂载即调 `pty_spawn`;`term.onData` → `pty_write` 被调用(spy)。
- L3:把捕获到的 pwsh 输出喂进 `@xterm/headless` → serialize → 断言提示符文本存在。
- L1:注入 profile 后从输出字节解析出 OSC 7 的 cwd 与 venv 自定义序列(对构造样本)。
  **验收 — 人工**:真机敲命令输出正确;改窗口大小内容重排正常。

#### 1.2 打开并编辑文件

**开发内容**:`panels/editor/`(CodeMirror 6,纯文本可编辑)、`fs/`(`fs_read_file/write_file`)、`ipc/fs.ts`。
**注意事项**:文件读写经后端,前端不碰 FS;保存写盘后才是 claude 能看到的版本。
**验收 — 自动化**:L1 fs 读写往返(临时目录);L2 编辑器载入内容、保存时以编辑后文本调 `fs_write_file`。
**验收 — 人工**:打开→编辑→保存→外部打开确认内容正确。

#### 1.3 多页签管理

**开发内容**:Dockview 标签/分组;新建终端/打开文件 = `api.addPanel` 加到焦点分组之后;关掉最后一个 → 回到”打开终端”空白页。
**注意事项**:面板只能来自 `panelRegistry`。
**验收 — 自动化**:L2:addPanel 在焦点分组加标签;关最后一个面板回到空白态(组件测试)。
**验收 — 人工**:标签切换/关闭的视觉行为。

#### 1.4 拖拽分屏 + 嵌套 + 调比例

**开发内容**:启用并配置 Dockview 内置 DnD(按光标悬停分组 + 靠近边决定方向)、嵌套、分隔条调比例、拖空自动合并。
**注意事项**:DnD 手感难自动化,重点放在布局结构正确性上。
**验收 — 自动化**:L2:用 API 构造分屏布局 → `toJSON` 断言结构 → `fromJSON` 还原一致(布局序列化往返)。
**验收 — 人工**:拖到四边各自正确分屏;嵌套;分隔条拖动调比例;拖空合并。(**主要人工**——交互手感)

#### 1.5 跑 claude

**开发内容**:确保 xterm 正确渲染 claude TUI;键位编码(Shift+Tab、Ctrl 组合)正确送达 PTY。
**注意事项**:**webview 键盘/IME 是 B 方案唯一可能撞墙处**——本步必须把中文 IME、Shift+Tab、Ctrl+C 当首要验收项;高级键(如完整 Kitty 键盘协议)可能受限。
**验收 — 自动化**:L3:喂录制的 claude 输出样本(备用屏、256/truecolor、光标移动)→ 断言缓冲/单元格/颜色正确(黄金快照);键位单测:Shift+Tab / Ctrl+C 产生正确字节序列。
**验收 — 人工**:真跑 `claude`——TUI 外观正确;Ctrl+C 取消;Shift+Tab 切 plan mode;**中文 IME 输入正常**;颜色正确。

### Phase 2 — 侧边栏与项目结构(M2)

**开发内容**:L1 项目(文件夹选择器手动添加、按项目分组)→ L2 主工作区 + worktree(`git worktree list`)→ L3 操作页面(各自独立 Dockview 布局,经 `layoutSerde` 存取;新建页面开 pwsh cwd 到路径);git 检测;worktree 管理(`git worktree add` + 命名 → 新操作页面 cwd 到该 worktree)与删除(警告 + 二次确认 + 关页签);页签按目录绑定 worktree。
**注意事项**:重启**不**恢复页签(项目与操作页面结构仍持久化,见架构"配置持久化");worktree 创建走 shell `git worktree add`、落 `.claude/worktrees/<name>`、分支 `worktree-<name>`(对齐 claude `--worktree`)并入 `.gitignore`,新页签里你自己敲 claude;`.claude/worktrees/` 在浏览器里隐藏(Phase 3 一并处理);可经 `WorktreeCreate` hook 感知 claude 自建的 worktree。
**验收 — 自动化**:

- L1:对临时 git 仓库测 git 检测、worktree `list/add/remove`;删除带页签时返回需确认信号。
- L2:项目/工作区/worktree/操作页面树的 store 状态正确;切操作页面 = 换布局 JSON(往返一致)。
  **验收 — 人工**:侧边栏树交互;切页面整套布局切换;新建 worktree 在正确目录开出页面。

-----

### Phase 3 — 文件浏览器与编辑器增强(M3)

**开发内容**:次级侧栏文件浏览器(跟随当前工作区;双击在焦点操作页面开编辑器面板;隐藏 `.claude/worktrees/`);CM6 语法高亮;行内 git diff 边栏(HEAD 基线,JetBrains 暗色行色);浏览器文件名 git 状态着色(JetBrains 暗色文件色);查找(Ctrl+F);外部改动处理(claude 改了已打开文件:干净自动重载、脏弹窗选择)。
**注意事项**:配色一律用 `theme/colors.ts` 的 token,禁止硬编码;CM6 比 Monaco 坑少,别中途换 Monaco(若换,注意”假设 Electron 环境”的 import 顺序等坑)。
**验收 — 自动化**:

- L1:git `status/diff` 对临时仓库正确。
- L2/L3:`gitGutter` 把 git hunk 映射到**正确的行色 token**(快照);文件名状态 → **正确颜色 token**(把”肉眼比色”变成断言);外部改动逻辑(干净 vs 脏分支)单测(模拟 change 事件 + mock store)。
  **验收 — 人工**:配色与 JetBrains 肉眼一致;diff 边栏观感;重载/弹窗交互。

-----

### Phase 4 — git / Python 状态栏(M4)

**开发内容**:底部状态栏——分支、diff/dirty 概要、Python 虚拟环境感知与展示。
**注意事项**:状态栏数据经后端;cwd 取自终端 OSC 7(Phase 1 profile 已提供);venv 取**当前激活**环境——经 Phase 1 profile 的自定义 OSC 序列回传 `$env:VIRTUAL_ENV`/`$env:CONDA_DEFAULT_ENV`(宿主读不到子 shell env,不能只查目录 `.venv`)。
**验收 — 自动化**:L1:分支名、ahead/behind、dirty 计数;venv 解析(对构造的自定义 OSC 序列样本,断言提取出激活环境名);L2:store → 状态栏渲染正确。
**验收 — 人工**:真实仓库/venv 下状态栏与实际一致。

-----

### Phase 5 — Claude Code 集成(M5)

**开发内容**:向全局 `~/.claude/settings.json` 装 `type:http`、`async` 的 hook(POST 事件到本应用本地 HTTP 服务,127.0.0.1 + 随机端口 + token;以 `SLTERM_PANE_ID` 存在为门);经 hook header 回传的 `SLTERM_PANE_ID` 建立**会话↔页签**精确映射;**锁存状态机**驱动终端页签状态点(`UserPromptSubmit`/`PreToolUse` → 运行中 → `Stop` → 完成;`PermissionRequest`/`Elicitation` → 需介入;`idle_prompt` → 等输入;`SubagentStop` 不算完成);操作页面节点聚合(需介入 > 运行中 > 空闲);桌面通知(需介入/完成;仅对应页签未聚焦时弹;点击跳转);提供 statusLine 脚本(仅在 claude TUI 显示);历史会话(读 `~/.claude/history.jsonl` 按项目过滤列标题/时间,`--resume <id>` 在该会话原 cwd 执行)。
**注意事项**:claude 是外部进程、CI 里难直接跑——用**录制的 hook 事件负载 / transcript JSONL 样本**驱动测试;通知”未聚焦才弹”的判定要单独可测;`--resume` 必须在会话原 cwd(项目/worktree 目录)执行,否则报 "No conversation found";hook 写全局 settings 须可逆;claude 未安装时优雅提示。
**验收 — 自动化**:

- L1/L2:状态映射(`UserPromptSubmit`/`PreToolUse` → 运行中、`PermissionRequest`/`Elicitation` → 需介入、`idle_prompt` → 等输入、`Stop` → 完成、`SubagentStop` **不**翻完成)对样本事件正确;`pane_id` 归属(同 cwd 多会话不串台)单测;聚合逻辑(页面 = 子项最高优先级)单测;通知门控函数(未聚焦才发)单测;历史会话样本(`history.jsonl`)→ 列表正确、`--resume <id>` 命令与 cwd 正确;statusLine 脚本对样本 JSON 输出含上下文%/成本(并兜底 `current_usage` 为 null)。
  **验收 — 人工**:真跑 claude,触发一次授权和一次完成 → 状态色正确、桌面 toast 实际弹出(且仅未聚焦时)、点击跳转;statusline 显示实时数字;resume 真能续上。

### Phase 6 — 体验完善(M6)

**开发内容**:图片粘贴(剪贴板有图 → 存临时文件 → 路径写入终端 `pty_write`);自定义键位(仅终端/UI 动作,其余透传 claude);暗色主题/字体可配置;性能打磨(合批渲染、WebGL、大输出)。
**注意事项**:图片粘贴在 Windows 上以”存临时文件 + 写入路径文本”为准;键位只管 UI 动作,别拦截要透传给 claude 的键。
**验收 — 自动化**:

- L1:剪贴板图 → 存临时文件返回路径(mock 剪贴板);L2:路径插入逻辑、键位动作分发表;L3/性能:向 `@xterm/headless` 灌大输出突发,断言处理耗时/不挂起 + 合批单测。
  **验收 — 人工**:把真实截图粘进 claude 并确认 claude 收到;自定义键位手感;大 `cat` 不卡;字体/主题切换。

## 三、贯穿全程的约束与持续验收

- **架构约束**(出自架构文档第七节,违反即视为不通过):前端只经 `ipc/` 通信;后端按功能分模块、共享只经 `AppState`;命令统一注册且返回 `AppError`;阻塞 I/O 用 `spawn_blocking`;DTO 双边对应(snake_case↔camelCase);面板只来自 `panelRegistry`;配色单点(`theme/colors.ts`);布局单点(`layoutSerde`);会话单点(`stores/sessions.ts`);`#[cfg(windows)]` 只在 `pty/spawn.rs`、`pty/shell.rs` 等处;capabilities 最小权限。
- **持续验收门**:每个 PR 必须 L1–L3 全绿 + 关键 L4 通过;新增命令必须带 L1 测试并在 capabilities 放行;新增面板必须注册并带 L2 测试。
- **回归基线**:claude 输出黄金快照、布局序列化样本、配色 token 断言纳入回归集,改动渲染/布局/配色时必须更新并复核。
- **每阶段完成定义(DoD)**:开发内容完成 + 该阶段自动化验收全绿 + 人工验收清单逐条勾选 + 不破坏既有回归集。

> 一句话:**自动化兜底正确性(L1–L4 + 日志),人工只把关机器够不到的”观感与手感”;每个阶段都先写好它的验收再算完成。**
