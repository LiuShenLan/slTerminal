# Phase 0 — 工程与测试基建(开发文档)

> 目标:把项目骨架和”自动化验收”所依赖的测试基建一次性立起来,让后续每个 phase 一开工就有 L1–L4 可用。配套:《完整开发计划》(方法学表)《技术架构设计文档》(路径划分与约束)。

## 1. 目标与范围

- 交付:可构建运行的空壳 Tauri 应用(暗色空 Dockview)+ 完整测试与 CI 基建。
- 不在范围:任何业务功能(终端、编辑器、侧栏等)。

## 2. 前置依赖

- 无(起点)。本机需:Node 20+、Rust 稳定版;Windows 上 VS C++ Build Tools + WebView2。

## 3. 涉及的模块与文件(铺骨架)

- 前端 `src/`:`main.tsx`、`App.tsx`,以及空目录 `ipc/ types/ stores/ workspace/ panels/ features/ theme/ lib/`(各放占位/index)。
- 后端 `src-tauri/src/`:`main.rs`(仅 `app_lib::run()`)、`lib.rs`、`error.rs`(AppError)、`state.rs`(AppState 空壳)、空模块 `pty/ fs/ git/ claude/ notify/`(各 `mod.rs` 占位)。
- 配置:`tauri.conf.json`、`capabilities/default.json`(最小)、`Cargo.toml`、`build.rs`。
- 测试:`src-tauri/tests/`(集成)、`src/**/*.test.ts(x)`(Vitest)、`test/terminal/`(headless 终端)、`e2e-tests/`(wdio.conf + specs)、`.github/workflows/ci.yml`。

## 4. 开发内容(分任务)

1. `npm create tauri-app`(React+TS+Vite)初始化;窗口暗色;`App.tsx` 渲染填满窗口的暗色 `DockviewReact`(空)。
1. 后端:`AppError`(thiserror + Serialize);`AppState` 空壳并 `manage`;`lib.rs` 的 `generate_handler!`(暂放一个 `ping`);接入 `tracing`。
1. 装依赖:前端 `dockview @xterm/xterm @xterm/addon-webgl @xterm/addon-fit @xterm/headless @xterm/addon-serialize`(termless 0.x 可选)、`vitest @tauri-apps/api`;后端 `portable-pty git2 notify`(占位,后续用)。
1. 测试基建:
- L1:`cargo test`,放一个样例(AppError 序列化)。
- L2:Vitest + `@tauri-apps/api/mocks`,放一个 `mockIPC` 样例(mock `ping`)。
- L3:`@xterm/headless` + serialize,放一个样例(feed 文本 → serialize 断言)。
- L4:`e2e-tests/` 配 `tauri-driver` + WebdriverIO,Windows 用 msedgedriver;一个样例(启动应用、断言标题)。
1. CI:GitHub Actions `windows-latest`,跑 `cargo test`、`npm test`、L3、`tauri build --debug`、关键 L4。

## 5. 技术要点与实现要领

- `main.rs` 不写逻辑,只 `app_lib::run()`;真实代码在 `lib.rs`。
- 约束守卫:加 ESLint 规则禁止 `@tauri-apps/api/core` 的 `invoke` 出现在 `src/ipc/` 之外(可 lint 强制)。注意"命令一律返回 `Result<_, AppError>`""阻塞 I/O 用 `spawn_blocking`"**无法 lint**,只能靠约定 + code review 把关。
- E2E:`wdio.conf` 的 `onPrepare` 跑 `tauri build --debug`,`beforeSession` 起 `tauri-driver`;capability `browserName: "wry"`,`tauri:options.application` 指向构建出的 exe;CI 装 msedgedriver 并与 WebView2 版本同步。
- `@tauri-apps/api/mocks` 在 jsdom 跑,快且隔离;事件/Channel 也能用回调模拟。

## 6. 注意事项与坑

- 锁工具链版本(Rust、Node、msedgedriver),CI 与本地一致。
- Playwright 在 Tauri 用不了(非 Chromium);E2E 只走 WebDriver。
- E2E 较脆,只覆盖关键流程;主力在 L1–L3。

## 7. 验收

### 7.1 自动化验收

- `cargo test` 通过(含 AppError 序列化样例)。
- `npm test`(Vitest)通过(含 mockIPC 样例)。
- L3 样例:feed `"hi"` → serialize 结果含 `hi`。
- `tauri build --debug` 成功;L4 样例:应用启动且窗口标题断言通过。
- CI 在 windows-latest 全绿。

### 7.2 人工验收

- 启动后出现暗色空 Dockview 窗口,无报错。

## 8. 完成定义(DoD)

- 骨架目录与四层测试各有可运行样例;CI 绿;约束守卫(invoke lint)生效。
