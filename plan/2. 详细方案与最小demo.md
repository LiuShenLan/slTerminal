# Scheme B 详细方案 + 最小 Demo 计划

> 原生核心(Tauri 2 + Rust)+ Web UI。目标:先做出一个最小可跑的 demo(一个窗口 + 统一分屏页签区 + 能开 PowerShell),再按既定的原子步骤往上加。

## 一、技术栈(选定)

|层       |选型                                                                     |说明                                                                               |
|--------|-----------------------------------------------------------------------|---------------------------------------------------------------------------------|
|外壳      |**Tauri 2**                                                            |Rust 核心 + Windows 上用系统 WebView2,不打包 Chromium(体积 ~个位数 MB,而非 Electron 的数百 MB)      |

## 二、进程与模块边界(两进程模型)

```
┌───────────────────────────────┐     invoke() 命令      ┌────────────────────────────┐
│  Web 前端 (WebView2)           │ ───────────────────▶  │  Rust 后端 (src-tauri)      │
│  - Dockview 布局               │                        │  - PtyState:               │
│  - 每个面板 = 一个组件:        │ ◀───────────────────  │    RwLock<HashMap<Id,Sess>>│
│      · Terminal(xterm.js)      │   Channel<PtyEvent>    │  - portable-pty 会话        │
│      · Editor(CodeMirror 6)    │   (输出流)             │  - 后续: fs / git / 通知    │
│  - 不直接碰 FS / 进程          │                        │  - 拥有所有 OS 访问         │
└───────────────────────────────┘                        └────────────────────────────┘
```

- **Rust 后端**:拥有一切 OS 访问(PTY、文件、git、进程、通知)。会话状态放一个 `RwLock<HashMap<SessionId, Session>>`。对外暴露 Tauri 命令(动作)+ 通过 `Channel<PtyEvent>` 推送 PTY 输出。
- **Web 前端**:纯 UI。Dockview 管布局,xterm.js / CodeMirror 是面板内容。**绝不**直接碰文件或进程,一切走 `invoke()`。
- **关键纪律**:每个 xterm.Terminal 实例创建一次、随标签存活;切换标签是隐藏 canvas,不销毁会话(切走再回来,PTY 还在跑)。

## 三、数据流(以一个终端为例)

1. 前端在 Dockview 里新建一个 Terminal 面板 → `invoke("pty_spawn", { shell, cwd, cols, rows })`。
1. 后端用 portable-pty 起 PowerShell,存进 PtyState,返回 `id`,并为这个会话开一个 `Channel<PtyEvent>`。
1. 后端读循环(一个线程):不断从 PTY 读字节 → 经 Channel 发 `Output { bytes }` 给前端。
1. 前端 `channel.onmessage` → `term.write(bytes)`。
1. 前端 `term.onData(input)` → `invoke("pty_write", { id, data: input })` → 后端写入 PTY。
1. 容器尺寸变化 → fit addon 算出 cols/rows → `invoke("pty_resize", { id, cols, rows })` → 后端 `pty.resize`。
1. 子进程退出 → 后端发 `Exit { code }` → 前端标记该标签。

## 四、IPC 契约(最小集合)

**命令(前端 → 后端,`invoke`)**

- `pty_spawn({ shell, cwd, cols, rows }) -> { id }`
- `pty_write({ id, data })`
- `pty_resize({ id, cols, rows })`
- `pty_kill({ id })`

**事件(后端 → 前端,推荐用 Tauri `Channel<PtyEvent>` 而非全局事件:有序、单会话定向、开销小)**

- `PtyEvent::Output { bytes }`
- `PtyEvent::Exit { code }`

> 这份契约就是 B 方案的核心架构产物。后面所有功能(开文件、git、claude 状态…)都是往后端加命令、往前端加面板,边界不变。

## 五、Windows 关键坑(先知道,免得被咬)

1. **ConPTY 自动解析**:portable-pty 的 `native_pty_system()` 在 Windows 上就是 ConPTY,调用处零 `#[cfg]`。
1. **spawn 串行化**:并发 spawn 会让某个 PTY 的输出管道卡死 → 用一把 `SPAWN_LOCK` 互斥锁把 spawn 串起来(你要开很多会话,这条必踩)。
1. **cwd 反斜杠**:传给 ConPTY 前把 cwd 规范化成 `\`(CreateProcessW 对 `/` 行为异常)。
1. **shell 启动**:`pwsh -NoLogo -NoExit -ExecutionPolicy Bypass -File <profile.ps1>`;优先级 `pwsh.exe`(PS7+)→ `powershell.exe`(5.1)→ `cmd.exe`。
1. **cwd / 命令边界 / venv 跟踪**(**Phase 1 即落地**,供 Phase 2/4/5 复用,不留到后期):portable-pty 在 Windows 上不返回 cwd → spawn 时注入一个 PowerShell profile,让它随 prompt 发 **OSC 7**(cwd)、**OSC 133 A/B/C/D**(提示符边界 + 退出码),以及一段**自定义 OSC 序列**回传当前激活的 venv/conda(`$env:VIRTUAL_ENV` / `$env:CONDA_DEFAULT_ENV`);宿主据此跟踪 cwd、命令边界与 venv,不解析提示符。
1. **claude 会话↔页签关联埋点**(Phase 1 埋点、Phase 5 使用):每个 PTY spawn 时注入唯一环境变量 `SLTERM_PANE_ID`(claude 及其 hooks 会继承)→ Phase 5 的 hook 回传该值,宿主据此把 claude `session_id` 精确绑定到对应终端页签,避免同 cwd 多会话串台。

## 六、最小 Demo(对应 M1.1:窗口 + 分屏页签区 + 开 PowerShell)

**范围**:一个暗色窗口,里面是 Dockview 的可停靠标签区;点一下能开出一个真正的 PowerShell,能输入、能看到输出。

**步骤**

1. `npm create tauri-app`(React + TS + Vite),窗口设暗色。
1. **后端**:加 `portable-pty`;实现 `pty_spawn / pty_write / pty_resize / pty_kill` 四个命令 + 一个读循环把输出经 `Channel<PtyEvent>` 推出去;`PtyState = RwLock<HashMap<Id, Session>>`;spawn 串行化;cwd 规范化为 `\`。
1. **前端**:加 `dockview`、`@xterm/xterm`、`@xterm/addon-webgl`、`@xterm/addon-fit`。根组件 = 一个填满窗口的 `<DockviewReact className="dockview-theme-dark">`。注册一个 `terminal` 面板组件:创建 `xterm.Terminal` → 挂 webgl + fit addon → 调 `pty_spawn` → 把 channel 的输出 `term.write`、把 `term.onData` 经 `pty_write` 写回 → 用 `ResizeObserver` 触发 fit 再 `pty_resize`。
1. 一个按钮/快捷键:`api.addPanel({ component: "terminal" })`。

**核心接线(示意)**

```ts
// 前端,terminal 面板内
const term = new Terminal();
term.loadAddon(new WebglAddon());
const fit = new FitAddon(); term.loadAddon(fit);
term.open(el);
const { id } = await invoke("pty_spawn", { shell: "pwsh.exe", cwd, cols: term.cols, rows: term.rows });
const ch = /* 该会话的 Channel<PtyEvent> */;
ch.onmessage = (e) => { if (e.type === "Output") term.write(e.bytes); };
term.onData((d) => invoke("pty_write", { id, data: d }));
new ResizeObserver(() => { fit.fit(); invoke("pty_resize", { id, cols: term.cols, rows: term.rows }); }).observe(el);
```

**参考项目**

- `marc2332/tauri-terminal` —— 最小可读的 Tauri + xterm.js + portable-pty 示例,直接当起点。
- `crynta/terax-ai`(Terax)—— 与你目标几乎一致的完整实现,遇到具体问题翻它的 `TERAX.md` 和源码。
- 想给第一发提速:`Tnze/tauri-plugin-pty`(`tauri-pty`)提供类 node-pty 的 JS `spawn()`,可暂时省掉手写 IPC。但建议正式版自己拥有 pty 命令层(后面的状态跟踪等都要靠它)。

## 七、紧接着的步骤(怎么落到这套栈上)

- **M1.2 开文件并编辑**:新增 `editor` 面板组件 = CodeMirror 6;后端加 `fs_read_file / fs_write_file`。

> 提醒:每个”操作页面”= 一套独立的 Dockview 布局,正好用 `toJSON/fromJSON` 序列化;切换操作页面 = 换一份布局 JSON。

## 八、风险与缓解

- **webview 键盘 / IME(B 方案唯一可能撞墙处)**:有人做 agent 终端评估 Tauri 后放弃,因为 webview 对多路复用器要的底层键盘/IME/输入抓取太窄。对你:① 你要敲中文,IME 合成要**尽早测**;② Shift+Tab、Ctrl 组合键用 xterm.js 的 `attachCustomKeyEventHandler` 接管;③ 完整 Kitty 键盘协议等少数高级特性在 webview 里可能受限、需额外工作。但你只是跑 claude 的 TUI(不是需要全局输入抓取的多路复用器),大概率没问题——只是把 IME + 关键键位作为 demo 之后的第一项验收。
- **GPU 是”web 意义上的”**:xterm.js WebGL + WebView2 合成器,够好但非 Zed 级;驱动有问题时切 canvas 回退(一个开关)。
- **若改用 Monaco**:注意”假设 Electron 环境”的坑(import 顺序、worker 等);CM6 基本避开。
