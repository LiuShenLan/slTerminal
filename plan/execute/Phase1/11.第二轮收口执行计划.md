## 4. 各 agent 任务卡

##### Agent G1a — onReady StrictMode 守卫（Phase '实现'，sonnet）

| 项目 | 内容 |
|------|------|
| **任务** | Workspace.tsx `onReady` 回调加 `useRef` 守卫，跳过 React StrictMode 第二次触发 |
| **文件** | `src/workspace/Workspace.tsx` |
| **操作** | ① React import 加 `useRef` ② Workspace 组件内第一行加 `const isReadyRef = useRef(false);` ③ `onReady` 回调首行加 `if (isReadyRef.current) return; isReadyRef.current = true;`（在 `window.__dockviewApi = api` 之前） |
| **验收** | `grep 'isReadyRef' src/workspace/Workspace.tsx` 有匹配；`npm test` workspace 测试 PASS；`npm run wdio` PASS |

> **修复机理解释**（Dockview 6.6.1 源码确证）：React StrictMode 下 `DockviewReact` 的 `useEffect` 经历 mount→unmount→remount。每次 mount 创建全新 `DockviewComponent`，unmount 执行 `api.dispose()` 完整清理。`useRef` 初始化器只运行一次，在 remount 时 `isReadyRef.current` 保持 `true`——跳过第二次 `onReady` 的 `addPanel`。**不使用 `useState`+`useEffect` 解耦**（增加不必要的重渲染），`useRef` 守卫是最简方案。

##### Agent G1b — pty_kill async + spawn_blocking（Phase '实现'，sonnet）

| 项目 | 内容 |
|------|------|
| **任务** | `pty_kill` 改为 async 命令，提取 session 后释放 RwLock 写锁，`spawn_blocking` 中执行阻塞操作 |
| **文件** | `src-tauri/src/pty/spawn.rs` |
| **操作** | ① `pty_kill` 函数签名改为 `pub async fn pty_kill(...)` ② session 提取与锁释放分离：`let session = { let mut sessions = ...; sessions.remove(...)? };`（RwLock 写锁在此 scope 结束时释放）③ `tokio::task::spawn_blocking(move \|\| { let mut child = session.child.lock().unwrap(); let _ = child.kill(); if let Some(handle) = session.reader_handle.take() { let _ = handle.join(); } // session drop 触发 ClosePseudoConsole，在 blocking 线程执行 }).await?` ④ 移除函数体中的 `mut sessions` 和 `mut session` 声明（session 不再需要 mut） |
| **验收** | `cargo check` 通过；`cargo test` 全部 PASS（PTY 集成测试 4 项）；无 clippy 警告 |

> **修复机理**：
> 1. `child.kill()` 本身不阻塞（`TerminateProcess`）——但 `handle.join()` 等待 reader 线程退出会阻塞。`session` drop 时 `pair.master`（含 HPCON）析构 → portable-pty 0.9.0 Drop 调用 `ClosePseudoConsole`——在 pre-Win11 24H2 上阻塞等待 conhost 退出。
> 2. 旧代码在持有 `RwLock` 写锁时执行上述阻塞操作 → 所有后续 `pty_kill`/`pty_write`/`pty_resize` 排队等锁 → 级联卡死。
> 3. 改为 async 后：提取 session（锁内，<1ms）→ 释放锁 → spawn_blocking 中 kill+join+drop（阻塞线程池线程，不阻塞 IPC worker 和后续命令）。
> 4. `State<'_, AppState>` 不 move 进 spawn_blocking——只 move 已提取的 `session` （owned 数据），`State` 借用已在锁释放后结束。

##### Agent G3 — 另存为对话框（Phase '实现'，sonnet）

| 项目 | 内容 |
|------|------|
| **任务** | `useCodeMirror.ts` `handleSave` 无 `filePath` 时弹出"另存为"对话框；集成 `tauri-plugin-dialog` |
| **文件** | `src/panels/editor/useCodeMirror.ts`、`src-tauri/Cargo.toml`、`src-tauri/src/lib.rs`、`package.json`、`src-tauri/capabilities/default.json` |
| **操作** | ① `npm install @tauri-apps/plugin-dialog@^2` ② `Cargo.toml` `[dependencies]` 加 `tauri-plugin-dialog = "2"` ③ `lib.rs` 在 `.plugin(tauri_plugin_opener::init())` 后加 `.plugin(tauri_plugin_dialog::init())` ④ `capabilities/default.json` `permissions` 数组加 `"dialog:default"` ⑤ `useCodeMirror.ts`：import `{ save } from "@tauri-apps/plugin-dialog"`；`handleSave` 改为 `async`：`if (!path) { const selected = await save({ defaultPath: "Untitled.txt", filters: [{ name: "所有文件", extensions: ["*"] }] }); if (!selected) return; path = selected; filePathRef.current = path; }` 然后执行 `fs.writeFile(path, content)` |

##### Agent G4 — dragDropEnabled: false（Phase '实现'，sonnet）

| 项目 | 内容 |
|------|------|
| **任务** | `tauri.conf.json` 窗口配置加 `"dragDropEnabled": false`，恢复 WebView2 HTML5 DnD 事件传递 |
| **文件** | `src-tauri/tauri.conf.json` |
| **操作** | ① `app.windows[0]` 对象中加 `"dragDropEnabled": false`（与 `"title"`/`"width"`/`"height"` 同级） |
| **验收** | `grep '"dragDropEnabled": false' src-tauri/tauri.conf.json` 有一次匹配；`npm run tauri build -- --debug --no-bundle` 构建成功 |

> **修复机理**：Tauri 默认 `dragDropEnabled: true` 在 WebView2 宿主层注册 OLE `IDropTarget` 并拦截所有 HTML5 DnD 事件。Dockview `auto` 模式下鼠标走 HTML5 后端——`dragstart/dragover/drop` 永远不到达 JS 层。Pointer 后端 `touchOnly=true` 忽略鼠标。结果：鼠标拖拽无可用后端。源码确证：Dockview `tab.js` 设 `this._element.draggable = caps.html5`，`resolveDndCapabilities` 在 `auto` 桌面端返回 `{ html5: true, pointer: true, pointerHandlesMouse: false }`。设为 `false` 后 Tauri 不注册 `IDropTarget`，HTML5 事件正常传递到 JS。

## 7. 注意事项

- **G1a 和 G1b 独立但互补**：G1a 修 `onReady` 双发导致的 Dockview 状态异常；G1b 修 `pty_kill` 同步阻塞导致的 RwLock 级联卡死。两者同时存在时，关闭面板卡死由 G1b 主导——G1a 单独修复无法消除 RwLock 锁争用
- **G1b `State` 借用安全**：`State<'_, AppState>` 只在提取 session 时借用（同步代码块内），不 move 进 `spawn_blocking`。`session` 是 owned 数据（`sessions.remove()` 返回），满足 `'static` 约束
- **G1b 非 Windows 平台**：`spawn_blocking` 在 Linux/macOS 同样适用——`handle.join()` 和 PTY master drop 在所有平台上都可能阻塞
- **G3 `save()` 返回 `string \| null`**：用户取消时 `null` → 不执行写文件。选择路径后更新 `filePathRef.current` 以便后续 Ctrl+S 直接保存
- **G3 `tauri-plugin-dialog` 权限**：`"dialog:default"` 是便捷简写（含 `allow-message`/`allow-open`/`allow-save`）。如需最小权限可改用 `"dialog:allow-save"`，但 `default` 更常见且面向未来
- **G4 无副作用**：slTerminal 当前无任何文件拖放功能（无 `onDragDropEvent` 监听）→ `dragDropEnabled: false` 仅恢复 HTML5 DnD 通路，不影响现有功能
- **G2 无需独立修复**：G2（终端创建后不可用）是 G1a 的级联效应——Dockview 状态受影响后 PTY spawn 异常。G1a 修复后自然恢复。若人工验收时 G2 仍存在，需额外调查
