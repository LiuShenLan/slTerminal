## 一、关键发现（需修正）

### 1.1 F3 buildNumber 掩码错误：`& 0x00FF_FFFF` → `& 0xFFFF`（中等）

**问题**：收口计划 F3 操作第②步写 `nt_version::get().build & 0x00FF_FFFF`。

**交叉核验结果**（Cygwin/ReactOS/Wine 三重独立来源确认）：

`RtlGetNtVersionNumbers` 返回的 build 值高位包含非预期比特位（如 Win7 SP1 返回 `0xF0001DB1` 而非 `0x1DB1`）。社区标准掩码是 `& 0xFFFF`（保留低 16 位）：

| 来源 | 掩码 | 引用 |
|------|------|------|
| Cygwin `wincap.cc` | `& 0xFFFF` | commit `63f19278`："upper bits set for no apparent reason" |
| ReactOS `CORE-6611` | `& 0xFFFF` | Win7 SP1 实测 |
| Wine NT API 参考 | `& 0xFFFF` | `RtlGetNtVersionNumbers` 实现注释 |

当前 Windows 10/11 build 号范围 10240–26200，均在 16 位内。`0x00FF_FFFF`（保留 24 位）比 `0xFFFF` 宽松但无实际风险——**建议统一到社区标准 `& 0xFFFF`**。

**修正结论**：F3 操作第②步改 `& 0x00FF_FFFF` → `& 0xFFFF`。

**落点**：F3 操作第②步

> 🔍 **核查结论：✅ 审查正确。**
> 
> **证据链**（3 agent 交叉核验一致）：
> 1. **`nt_version` crate 源码**（docs.rs）：`get()` 直接透传 `RtlGetNtVersionNumbers` 原始返回值，**crate 内部不应用任何掩码**。调用者必须自行掩码。
> 2. **Geoff Chappell 权威文档**：低 16 位是真实 build 号，`0xF0000000`（高 4 位）是 free/checked build 指示符。
> 3. **Cygwin `wincap.cc` commit `63f19278`**：`version.dwBuildNumber &= 0xffff`——"upper bits set for no apparent reason"。
> 4. **ReactOS CORE-6611**：Win7 SP1 实测 `0xF0001DB1 → 0x1DB1 = 7601`，掩码 `& 0xFFFF`。
> 5. **Wine 源码 `ntdll/version.c`**：`*build = (0xF0000000 | versionInfo.dwBuildNumber)`——确认高 4 位由 API 设置。
> 6. 当前 Windows 10/11 build 号范围 10240–26200，均在 16 位内（< 65535）。`& 0x00FF_FFFF` 保留 24 位在实践中无害但不符合社区规范。
> 
> **结论：掩码应改为 `& 0xFFFF`。同时修正收口计划 §7 "build number 高 4 位为 `0xF`"→"32 位值的高 4 位为 `0xF0000000`"。**

### 1.2 F3 `nt_version` 依赖需要条件编译（中等）

**问题**：收口计划 F3 写 `Cargo.toml：`[dependencies]` 下加 `nt_version = "0.1"``。

**交叉核验结果**：`nt_version` crate 在 `lib.rs` 中包含 `#[cfg(not(target_os = "windows"))] compile_error!(...)`——非 Windows 目标编译硬失败。在 macOS/Linux CI 或 `cargo check --target x86_64-unknown-linux-gnu` 时构建中断。

**修正结论**：F3 的 `Cargo.toml` 改为条件依赖：
```toml
[target.'cfg(windows)'.dependencies]
nt_version = "0.1"
```
Rust 代码中 `get_windows_build_number` 命令加 `#[cfg(windows)]` 守卫，非 Windows 返回 fallback `21376`。

**落点**：F3 操作第①步

> 🔍 **核查结论：✅ 审查正确。**
>
> **证据链**：
> 1. **`nt_version` crate `lib.rs` 源码**（docs.rs）：第 8-10 行 `#[cfg(not(target_os = "windows"))] compile_error!("This crate is for querying Windows, but the target isn't Windows.");`——非 Windows 目标编译硬失败。
> 2. **crate 自身未在 Cargo.toml 中用条件依赖**——它用"编译时拒绝"策略而非"依赖解析时拒绝"，但这不影响下游项目必须加条件依赖的事实。
> 3. **Linux CI / `cargo check --target x86_64-unknown-linux-gnu`** 会触发编译错误，阻断跨平台 CI。
> 4. **命令函数同样需要 `#[cfg(windows)]` 守卫**——非 Windows 上 `get_windows_build_number` 应返回 fallback `21376` 或 `NotImplemented` 错误。
>
> **结论：审查建议正确。Cargo.toml 需 `[target.'cfg(windows)'.dependencies]`，Rust 代码需 `#[cfg(windows)]` 守卫。**

### 1.3 F1 onReady 自动创建面板与 Dockview 初始化竞态（中低）

**问题**：收口计划 F1 写 `onReady` 回调中调用 `api.addPanel(...)` 自动创建终端。

**交叉核验结果**（Dockview 源码分析）：

`DockviewComponent.addPanel()` 在最简形式（无 `position` 参数）下会将面板添加到内部默认 group。但如果 `onReady` 触发时 watermark 仍在渲染且没有活跃 group，可能触发：
- 无 position 参数时，`addPanel` 内部选择默认 group 或创建新 group
- Dockview 6.6 的默认行为：无现有面板时自动创建一个 group 并放入面板

**结论**：`onReady` 中直接 `addPanel` 通常安全，但建议加防御：
```typescript
const onReady = (event: { api: DockviewApi }) => {
    const { api } = event;
    // 自动创建终端面板（给用户一个立即可用的终端）
    api.addPanel({
        id: `terminal_${Date.now()}`,
        component: 'terminal',
        title: 'PowerShell',
        renderer: 'always',
    });
};
```
如果担忧 edge case，可用 `setTimeout(0)` 延迟到下一个微任务。

**落点**：F1 操作第②步

> 🔍 **核查结论：❌ 审查不成立。`setTimeout(0)` 不必要且有害。**
>
> **证据链**（Dockview 6.6 源码 + 官方文档双重确认）：
> 1. **Dockview 官方 README Quick Start 示例**（`node_modules/dockview-react/README.md` L57-78）：直接在 `onReady` 中调用 `addPanel`——无 `setTimeout`、无 `useEffect` 延迟。这是**官方推荐的模式**。
> 2. **`onReady` 触发时序**（`node_modules/dockview/dist/esm/dockview/dockview.js` L109-114）：`createDockview()` → `api.layout()` → `props.onReady({ api })`，全链路**同步执行**。watermark 在 `createDockview()` 构造函数内已同步渲染完成，`onReady` 回调时 watermark 已就绪。
> 3. **`addPanel` 无 group 时的处理路径**（`node_modules/dockview-core/dist/esm/dockview/dockviewComponent.js` L1592）：无现有面板时走 `else` 分支，在根位置 `[0]` 创建新 group 并放入面板——这是一个**明确定义、经过测试的代码路径**。
> 4. **`setTimeout(0)` 的副作用**：将 `addPanel` 推迟到下一微任务会导致 watermark 先渲染、一帧后再被 panel 替换——产生**肉眼可见的闪烁**。这是负优化。
>
> **结论：onReady 中直接 `addPanel` 完全安全，无需任何防御性延迟。审查建议应撤回。**

## 三、可优化建议（非阻塞）

### 3.1 F1 空态自动创建终端 vs "打开终端"空白页需求

**背景**：需求 §三.1 规定"新建操作页面 → 展示空白页，页上有一个"打开终端"按钮"。收口计划 F1 选择 onReady 自动创建终端——绕过空白页。

**分析**：这对 Phase 1 收口是务实的（自动验收路径最短），但与需求规格存在语义偏差。建议在 DoD 中标注此差异——Phase 2 引入操作页面概念后可恢复空白页逻辑。

### 3.2 F4 `-EncodedCommand` 脚本长度上限

**发现**：Base64 编码后长度约为原始脚本的 ~2.67 倍。`shell-integration.ps1` 目前 ~75 行（~2KB），编码后约 ~5.4KB——远未触及 Windows 命令行 32767 字符上限。当前安全，但未来若脚本显著增长需监控。

### 3.3 F4 AMSI 仍扫描解码后内容

**发现**：`-EncodedCommand` 消除了"写磁盘+执行"的文件 I/O 检测面，但 PowerShell 引擎在脚本编译前仍将解码后内容送入 AMSI `ScanContent()`。如果杀毒软件已有 `shell-integration.ps1` 的内容签名，无论通过 `-File` 还是 `-EncodedCommand` 执行都会命中。此发现不影响收口计划的技术方向——消除文件 I/O 仍是正向改进。
