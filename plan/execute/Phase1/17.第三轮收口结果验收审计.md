## 3. 根因详析

### 3.1 F1：`isDisposedRef` 不复位（定案根因，阻断所有 fit 路径）

**文件**：`src/panels/terminal/useXterm.ts`

```typescript
// L56
const isDisposedRef = useRef(false);     // 初始值 false

// L111-219：主 useEffect
useEffect(() => {
    if (!container || terminalRef.current) return;
    // isDisposedRef.current 从未被设为 false!  ← 缺陷

    // ...创建 Terminal, fitAddon, WebGL, term.open()...
    // ...fonts.ready.then(() => canFit(...))...  ← isDisposedRef=true → 跳过

    return () => {
      isDisposedRef.current = true;  // ← 仅在此处设为 true
      terminalRef.current = null;
    };
}, [container, ..., windowsBuildNumber, ...]);
//                ^^^^^^^^^^^^^^^^^^ 触发条件
```

**触发**：`windowsBuildNumber` 从 `undefined` → `number`（IPC 返回，10-50ms）→ 依赖变化 → cleanup 设 `true` → body 不复位 → 此后所有 `canFit()` 的 `isDisposedRef.current === true`。

**被阻断的所有调用点**：

| 调用点 | 行号 | 阻断后后果 |
|--------|------|-----------|
| `fonts.ready` fit | L158–159 | 首次渲染后不 resize |
| RO doFit (rAF) | L231–232 | 容器变化不 resize |
| setTimeout(0) | L261–265 | 回退全失效 |
| setTimeout(50) | L261–265 | 回退全失效 |
| setTimeout(200) | L261–265 | 回退全失效 |

### 3.2 F2：RO effect 闭包过时（终端重建后引死实例）

**文件**：`useXterm.ts:221-271`

G2 RO effect 仅依赖 `[container]`。`windowsBuildNumber` 变更不触发重跑：
- Cleanup 中旧 Terminal dispose，`terminalRef.current = null`，`fitAddonRef.current?.dispose()`
- Body 中新 Terminal 创建，`terminalRef.current` 指向新实例
- RO 闭包内 `term`/`fitAddon` 仍指旧 dispose 的实例
- `canFit(oldTerm, ...)` → 条件 4（`!terminal.element`）失败（dispose 后 element 为 null）

**结论**：即使修复 F1，RO 仍引死 Terminal → 终端重建后 resize 响应失效。

### 3.3 F3：`windowsBuildNumber` 驱发不必要的重建

F3 独立 effect（L273–283）已处理 `windowsBuildNumber` 异步更新：
```typescript
useEffect(() => {
    const term = terminalRef.current;
    if (term && windowsBuildNumber !== undefined) {
      term.options.windowsPty = { backend: "conpty", buildNumber: windowsBuildNumber };
    }
}, [windowsBuildNumber]);
```

但主 effect 依赖数组也含 `windowsBuildNumber`——IPC 返回时主 effect 先执行**销毁+重建**（覆盖 F3 的非破坏意图），F3 再在新 Terminal 上再次设置 `windowsPty`（冗余）。

**该依赖项是触发 F1/F2 的唯一条件**（无它则 effect 仅因 `container` 变更触发一次）。

### 3.4 P1：增量面板创建时 CSS 布局异步——effect 早于布局

**来源**：Round 2 #4/#6 失败分析 agent + dockview overlay 时序分析 agent。

dockview 使用三层异步：`queueMicrotask`（AsapEvent）→ `queueMicrotask`（overlay attach）→ `requestAnimationFrame`（位置/尺寸）。React 18 用 `MessageChannel`（宏任务）调度 effects。

事件循环顺序：**dockview 微任务 → React MessageChannel（宏任务）→ rAF → 浏览器布局 → 绘制**。

当增量创建面板（#4 Watermark / #6 header"+"）时：
- React 的 effect（`useXterm.mainEffect`）在 MessageChannel 宏任务中触发
- 此时 overlay 的 rAF 可能尚未执行——CSS `height:100%` 链可能未解析
- `canFit()` 条件 3（`offsetWidth===0`）可能失败
- Round 2 无条件 fit→尝试在 0×0 容器上 fit→`proposeDimensions()` 返回 NaN→fit 静默跳过

**Round 2 #1 能工作的原因**：初次渲染时整棵 DOM 树一起布局——首次 layout 一次性计算所有元素尺寸，effect 触发时容器尺寸已有效。

**Round 2 #7 能工作的原因**：右键→菜单→选择→action 的交互序列提供了额外的事件循环轮次（渲染菜单、关闭菜单），布局在此期间完成。

### 3.5 P2：PTY 在 80×24 创建——ConPTY 初始尺寸错误

**来源**：PTY 80×24 尺寸分析 agent。

```typescript
// TerminalPanel.tsx:38-39
cols: 80,      // ← hardcoded
rows: 24,      // ← hardcoded

// useXterm.ts:183-184 传给 pty.spawn
pty.spawn({ panelId, cols, rows }, handlePtyOutput)
```

portable-pty 0.9.0 已启用 `PSEUDOCONSOLE_RESIZE_QUIRK`（防止 buffer dump），但：
- pwsh 以 80 列格式化首个提示符
- Microsoft terminal #10400：`CreateProcess` 后立即 `ResizePseudoConsole` 可能被静默忽略
- 后续 `pty.resize()` 可能不生效——ConPTY 保持 80×24
- 即使 resize 成功，已有输出（首个提示符）不回流

### 3.6 P3：`pty.spawn()` 重复调用——5 个竞态问题

**来源**：pty.spawn 竞态分析 agent。

1. **cleanup 中 `pty.kill()` 未 await**（L210–211）：fire-and-forget，React cleanup 不能 async
2. **`SPAWN_LOCK` 仅保护 spawn**（spawn.rs:62）：kill 不获取锁，kill+spawn 可并发
3. **`sessionIdRef` 初始为 null**（L52）：若 re-render 早于首次 spawn resolve，`kill(null)` 被跳过→ 旧 PTY 永不终止
4. **两个 spawn 的 `.then()` 竞态覆盖 `sessionIdRef`**：旧 Promise 晚于新 Promise resolve→ 覆盖为新 ID 后又被旧 ID 覆盖
5. **Rust 端无 panel 级去重**：同一 panel 可同时有多个活跃 PTY 会话（不同 UUID）

### 3.7 P4：CSS `height:100%` 链在 `.dv-overlay-render-container` 断裂

**来源**：CSS height 100% 解析分析 agent。

dockview 面板 DOM 层级中：
```
.dv-content-container (flex-grow:1, 有有效高度)
  └── .dv-overlay-render-container (position:relative; ⚠️ 无 height!)
       └── .dv-render-overlay (position:absolute; height:100%)
```

`.dv-overlay-render-container` 仅设 `position: relative`，无高度属性。CSS 默认 `height: auto` = 内容高度 = **0**（唯一子元素 `.dv-render-overlay` 为 `position:absolute`，脱离正常流）。因此 `.dv-render-overlay` 的 `height:100%` = **100% of 0 = 0**。

dockview 通过 `overlayRenderContainer.attach()` → `queueMicrotask` → rAF 设置显式 px 高度修复。但经过 `watchElementResize`（1 帧 rAF）+ `debouncedUpdateAllPositions`（1 帧 rAF）共 **2 帧延迟**。React 的 `useEffect` 在此 2 帧窗口内触发——此时 `offsetHeight === 0`。

**证据**：
- `dockview.css:2236-2238`：`.dv-overlay-render-container` 仅 `position: relative`
- `overlayRenderContainer.js:105-143`：rAF 中设显式 px 尺寸
- `dom.js:17-28`：`watchElementResize` 内又包裹一层 rAF
- `dockviewComponent.js:1788-1796`：`debouncedUpdateAllPositions` rAF

**影响**：终端创建时容器 offsetHeight=0 → `canFit()` 条件3 失败（若 fonts.ready 未在之前 resolve）或 `proposeDimensions()` 返回 NaN/undefined → fit 静默跳过 → 与 F1（isDisposedRef）共同作用→终端永久 80×24。

### 3.8 B1：`-NoExit` 缺失——pwsh 执行完脚本退出

**文件**：`src-tauri/src/pty/shell.rs:36-43`

```rust
fn build_pwsh_command(pwsh: &str) -> CommandBuilder {
    let mut cmd = CommandBuilder::new(pwsh);
    cmd.arg("-NoProfile");
    cmd.arg("-NoLogo");
    // 缺少 cmd.arg("-NoExit");
    cmd.arg("-EncodedCommand");
    cmd.arg(encode_utf16le_base64(get_shell_integration_script()));
    cmd
}
```

**证据**：
- `git log --all -S "-NoExit" -- "*.rs"` → 零命中——**从未存在**
- 计划文档 `plan/2. 详细方案与最小demo.md:74` 明确要求 `-NoExit`
- 参考项目 terax-ai 正确实现：`-NoExit -File <profile>`
- `-EncodedCommand` 等价于 `-Command`——执行后默认退出

**影响**：pwsh 执行 shell-integration.ps1 → 脚本最后一行结束 → pwsh 退出 → 用户永无交互提示符。

**为什么 Round 2 #1/#7 标"通过"**：初始实现用 `-File`（commit `0d6707f`），后因 AMSI/ASR 杀毒软件误杀在 `b887b63` 切换为 `-EncodedCommand`。两种方式均缺 `-NoExit`。两条路径标"通过"可能是因为——测试仅验证面板创建成功+终端有正确尺寸，未实际验证交互式提示符。

### 3.9 P5：初始化顺序反置——`open→spawn(80×24)→fit` vs 业界标准的 `open→fit→spawn`

**来源**：全部 5 个参考终端（electerm、Hyper、Tabby、terax-ai、termoras）对比分析。

| 终端项目 | stars | 初始化顺序 | PTY spawn 尺寸来源 |
|----------|-------|-----------|-------------------|
| **electerm** | 11k | open → fit → spawn | `term.cols/rows`（fit 后真实值） |
| **Hyper** | 43k | open → fit → spawn | Redux store（fit 后） |
| **Tabby** | 59k | open → fit → ready → spawn | `fitAddon.proposeDimensions()` |
| **termoras** | — | open → rAF轮询→ fit → spawn | rAF 检查 `offsetWidth>0` 后 fit |
| **terax-ai** | 7.2k | open → visible → fit → spawn | 面板可见时 fit 后 |
| **slTerminal** | — | **open → spawn(80×24) → fit** | **硬编码 80×24** |

**全部 5 个参考项目无一使用硬编码尺寸 spawn PTY。全部先 fit 后 spawn。**

**termoras 的 rAF 轮询方案**（最接近 slTerminal 场景）：
```typescript
// 主动轮询 offsetWidth/offsetHeight，直到容器有真实尺寸
let connectAttempts = 0;
const connectPty = () => {
    const el = containerRef.current;
    if (!el || el.offsetWidth === 0 || el.offsetHeight === 0) {
        connectAttempts++;
        if (connectAttempts < 120) requestAnimationFrame(connectPty);
        return;
    }
    fitAddon.fit();
    pty.spawn({ cols: term.cols, rows: term.rows }, ...);
};
requestAnimationFrame(connectPty);
```
最多 120 帧（~2 秒），彻底避免了 CSS 布局异步和 ConPTY #10400 竞态。

## 5. 附带发现

### 5.1 `proposeDimensions()` vs `canFit()` 度量不一致
- `proposeDimensions()`：`getComputedStyle(parentElement).height/width`
- `canFit()`：`containerEl.offsetWidth/offsetHeight`
- 两者可能在不同时机给出不同结果（`getComputedStyle` 可能返回 `"auto"`）

### 5.2 E2E 测试假阳性
`e2e_marker_direct` 包含子串 `e2e_marker`——即使 PTY 零输出断言也通过。

### 5.3 `language-mapping.test.ts` 弱断言
19 个用例全用 `toBeInstanceOf(LanguageSupport)`——无法区分语言映射错误。

### 5.4 F3 独立 useEffect 是死代码
主 effect 已因 `windowsBuildNumber` 重跑——F3 的更新被主 effect 的销毁重建覆盖。

### 5.5 `fonts.ready` 微任务早于首次布局
CSS Font Loading L3 规范 + WebKit Bug 174030——`fonts.ready` 在字体缓存时以微任务 resolve，可能早于浏览器对新增 DOM 的首次布局。

### 5.6 xterm.js v6 `_isPaused`（IntersectionObserver）
`visibility:hidden` 时渲染延迟，变为可见时自动恢复。不影响 `terminal.resize()`。

### 5.7 Dockview overlay 渲染三层异步
`queueMicrotask`（AsapEvent）→ `queueMicrotask`（overlay visibilityChanged）→ `requestAnimationFrame`（位置/尺寸/visibility）

### 5.8 React 18 production 用 MessageChannel 调度 effects
非微任务——effects 确定性地晚于 dockview 的 queueMicrotask。

### 5.9 参考项目 terax-ai 的正确实现
[Tauri 2 + portable-pty 终端项目](https://github.com/crynta/terax-ai)：`-NoLogo -NoExit -ExecutionPolicy Bypass -File <profile>`——slTerminal 应遵循的模式。

### 5.10 ⚠️ xterm.js #5800：Vite 生产构建双重压缩崩溃

**来源**：Vite 构建分析 agent。

**发现**：xterm.js v6.0.0 发布**已压缩的 ESM**（`lib/xterm.mjs`，345KB 单行）。Vite 8 使用 rolldown SWC minifier 对已压缩代码**再次压缩**→标识符被二次混淆→`InputHandler.requestMode` 闭包内引用崩溃→`Uncaught ReferenceError: i is not defined`。

- 上游 Issue：[xtermjs/xterm.js#5800](https://github.com/xtermjs/xterm.js/issues/5800)
- 上游状态：截至 2026-06 **未修复**
- 影响：仅 `vite build`（dev 模式直接服务原始 ESM，无此问题）
- 当前症状不符（80×24 小终端 vs 白屏崩溃）→说明 slTerminal 的构建产物**可能暂未触发**，但风险存在

**验证**：构建产物 `dist/assets/index-CbIzkF66.js`（1667KB）已确认经过双重压缩。`requestMode` 函数参数被短路为 `e`。

**修复**：将 xterm 包标记为 external，不参与 Vite 打包（§7.10）。

### 5.11 WebGL 渲染器回退链确认安全

**来源**：WebGL 回退路径分析 agent。

- `new WebglAddon()` 构造失败 → DOM 兜底 ✓
- `activate()` 在 `open()` 前仅注册 `onWillOpen` 回调，不抛异常 ✓
- `onWillOpen` 中 `activate()` 失败 → xterm.js 核心 try/catch 捕获 → `hasRenderer()=false` → DOM 兜底 ✓
- `onContextLoss` → `dispose()` → 清理回调中 `setRenderer(DomRenderer)` → DOM 恢复 ✓
- `detectWebgl()` 使用独立临时的 canvas——与 WebglRenderer 内部创建的实际渲染 canvas **不同实例**，属性也不同（检测用 `failIfMajorPerformanceCaveat: true`，渲染用 `antialias: false`）

**结论**：渲染器回退链设计健全。`loadAddon` 在 `open()` 之前调用时不会失败。**非本次 bug 源。**

### 5.12 Tauri Channel 生命周期确认无泄漏

**来源**：Channel 生命周期分析 agent。

- JS 端 Channel 无 `close()`/`dispose()` 方法——**唯一清理路径**是 Rust 端 drop → 发送 `{end: true}` → `cleanupCallback()` → `unregisterCallback(id)`
- `pty_kill` 触发 reader 线程 EOF → Channel drop → `end` 消息 → 自动清理 ✓
- `onmessage` 是动态 getter——每次读取当前值，非构造时捕获
- 已知 Issue [tauri#13133](https://github.com/tauri-apps/tauri/issues/13133)：`onmessage` 未 delete 时可能阻止 GC——但 `end` 到达后自动解除

**结论**：Channel 生命周期在正常 PTY 关闭流程中正确。**非本次 bug 源。**

### 5.13 Loading 遮罩确认无干扰

**来源**：loading 遮罩分析 agent。

- `position:absolute; inset:0` 遮罩是 xterm.js 的**兄弟元素**（非祖先）→ 遮挡视觉但不影响布局
- 移除遮罩不触发 ResizeObserver→不影响 fit()
- `pointerEvents: "none"` 正确 → 点击穿透到 xterm textarea
- `transition: "opacity 0.3s"` 是**死代码**——React 条件渲染直接移除 DOM，transition 无法插值
