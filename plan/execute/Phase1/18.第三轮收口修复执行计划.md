## 一、背景

第三轮收口 5 项修复（G2/F3/D3/D4）自动化全部通过，但人工验收暴露 G2 终端仍不可用（80×24 小窗口、无 PowerShell 提示符）。

审计文档（17，六轮 40 路 agent）确证 4 个独立根因——3 个前端（F1/F2/F3）+ 1 个后端（B1）：

| # | 根因 | 严重度 | 位置 | 修复行数 |
|---|------|--------|------|---------|
| F1 | `isDisposedRef.current` 未在 effect body 复位——所有 fit 路径被阻断 | 🔴 | `useXterm.ts` | 1 行 |
| F2 | RO effect 仅依赖 `[container]`——终端重建后持死实例引用 | 🔴 | `useXterm.ts` | ~30 行 |
| F3 | `windowsBuildNumber` 在主 effect 依赖数组——触发不必要的终端销毁重建 | 🟡 | `useXterm.ts` | 1 行删 |
| B1 | `shell.rs` 缺少 `-NoExit`——pwsh 执行完脚本立即退出 | 🔴 | `shell.rs` | 1 行 |
| P1 | 增量创建面板时 `useEffect` 早于浏览器 CSS 布局 | — | 预发现象 | rAF 轮询覆盖 |

**损坏链**：`windowsBuildNumber` IPC 异步返回（10-50ms）→ 主 effect 依赖变化重跑 → cleanup 设 `isDisposedRef = true` → body 不复位 → `canFit()` 条件 5 永久 `false` → `fonts.ready` fit、RO doFit、setTimeout 回退全部阻断 → 终端停留在 80×24。

**D3/D4/G4 状态**：编辑器语法高亮 ✅ | dialog 架构合规 ✅ | 页签拖拽 ✅ | 终端可用 ❌ 阻断。

## 二、修复方案

### 2.1 useXterm.ts 重构（F1 + F2 + F3）

**文件**：`src/panels/terminal/useXterm.ts`

**核心思路**：用 rAF 轮询替代 ResizeObserver 作为首次 fit 的触发源（RO 仅用于后续 resize），彻底消除"effect 执行早于 CSS 布局"的时序窗口。

**变更清单**：

| # | 变更 | 说明 | 参见审计 |
|---|------|------|---------|
| 1 | `isDisposedRef.current = false` 置于 effect body 首行 | 解决 F1——cleanup 设 true 后 body 不复位的 bug | §3.1 |
| 2 | 主 effect 重写：`term.open()` 后 rAF 轮询 `offsetWidth > 0`（30 帧/~500ms 上限）→ `fitAddon.fit()` → 获取真实 cols/rows → `pty.spawn()` | 解决 P1——不等 ResizeObserver、不依赖 CSS 布局时序。超时回退 80×24 spawn | §3.1/§3.2 |
| 3 | RO 移回主 effect 内部——随 Terminal 重建自动重建 | 解决 F2——消除闭包过时引用 | §3.2/§3.3 |
| 4 | `windowsBuildNumber` 从主 effect deps 移除 | 解决 F3——IPC 异步返回后不再触发全量重建。F3 独立 effect 保留（动态更新 `windowsPty`） | §3.4 |
| 5 | `fonts.ready` fit 外包裹 rAF | 防御 P1——字体就绪后的 fit 也在下一帧执行 | §3.1 |
| 6 | PTY resize 加 100ms debounce | 防止拖拽窗口时 60 次/秒 resize 轰炸 ConPTY | §3.5 |
| 7 | 删除 G2 独立 RO effect | RO 已移回主 effect——无独立存在必要 | §3.3 |
| 8 | `canFit()` 保留（测试依赖 + fonts.ready/RO doFit 防御性守卫），rAF 轮询路径不依赖其条件 5（isDisposed） | 轮询自身检查 `offsetWidth > 0` 和 `terminal.element` | §3.1 |

**改动量**：~35 行净删增。

**验收**：

| 命令 | 期望 |
|------|------|
| `npx tsc --noEmit` | 退出 0 |
| `npx eslint src/panels/terminal/useXterm.ts` | 退出 0 |
| `npm test` | 退出 0，canFit 测试 PASS |

### 2.2 shell.rs 加 `-NoExit`

**文件**：`src-tauri/src/pty/shell.rs`

**根因**：`build_pwsh_command` 生成的 pwsh 命令行不含 `-NoExit`——执行完 `-EncodedCommand` 脚本后进程立即退出，用户看不到提示符。

**变更**：`build_pwsh_command` 函数，`-NoLogo` 后插入 `.arg("-NoExit")`。

参见审计 §4。

**改动量**：1 行。

**验收**：

| 命令 | 期望 |
|------|------|
| `cargo build --debug` | 退出 0 |
| `cargo test` | 退出 0，`test_shell_integration_script_embedded` PASS |

### 2.3 App.css 补全高度链

**文件**：`src/App.css`

**根因**：Dockview overlay 渲染容器（`.dv-overlay-render-container`）缺少 `height: 100%`——CSS 高度链在 overlay 层断裂 → 容器在前 2 帧高度为 0 → rAF 轮询需等待 2 帧才能检测到有效尺寸。

**变更**：文件末尾追加 3 行 CSS：

```css
.dv-dockview .dv-overlay-render-container {
  height: 100%;
}
```

参见审计 §5。

**改动量**：3 行。

**验收**：

| 命令 | 期望 |
|------|------|
| `npm test` | 退出 0 |

## 六、注意事项

1. **`canFit()` 保留不删**——被 L2 单元测试依赖（`canFit.test.ts`），且 `fonts.ready` 和 RO doFit 仍用它做防御性守卫
2. **F3 独立 effect 保留**——`windowsBuildNumber` 异步更新 `windowsPty` 选项，功能正确；只从主 effect deps 移除
3. **`isDisposedRef` 保留**——用于 cleanup 时阻止异步回调操作已销毁终端
4. **PTY resize debounce 100ms**——每 100ms 最多发一次 resize 到 ConPTY，参考 Tabby 模式
5. **rAF 轮询上限 30 帧（~500ms）**——正常情况 CSS 修复后 0-1 帧就绪；Dockview overlay 最多 2 帧；30 帧充裕
6. **超时回退 80×24**——极端情况（容器永不就绪）降级但不丢功能。后续 RO doFit 会纠正尺寸
7. **不修改 `TerminalPanel.tsx`**——硬编码 `cols:80, rows:24` 仅作 fallback，实际 PTY spawn 用 fit 后真实尺寸
8. **不修改 `canFit()` 导出签名**——测试文件依赖此导出
9. **`App.css` 全局生效**——所有 Dockview 面板（当前+未来）自动受益
