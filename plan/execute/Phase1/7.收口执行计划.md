| # | 问题 | 严重度 | 根因 | 修复方向 |
|---|------|--------|------|---------|

### 1.3 编排决策

按 `plan-generation.md` 决策树：F1 零共享文件、F2/F4 无文件冲突 → **F1∥F2∥F4 全并行**。F3 依赖 F2 签名（编译时）和 F4 的 Cargo.toml（文件冲突）→ F3 在 F2+F4 之后串行执行。

| 层 | 方式 | 内容 |
|----|------|------|
| `/goal` 主干 | 15 turns 循环 | 锁 F1–F4 全部完成 + L1–L4 全绿 |
| W0 前置核验 | parallel 2 agent | 读现状确认切入点 + 读 Dockview/xterm.js 源码交叉验证 |
| W1 实现 | parallel 3 + pipeline 1 | Phase 1: F1∥F2∥F4（全并行）→ F3（等待 F2+F4 完成后串行） |

## 7. 注意事项

- **F4 的 `base64` crate 已存在 Cargo.lock（0.22.1）**：通过 tauri-codegen/reqwest/wry 传递依赖，无需新下载。仅需在 Cargo.toml 显式声明
- **`nt_version` crate 内部使用 `RtlGetNtVersionNumbers`**：返回原始值，高 4 位为 `0xF0000000`（free/checked build 指示符）。需 `& 0xFFFF` 掩码取低 16 位真实 build 号（Cygwin/ReactOS/Wine 社区标准）。当前所有 Windows 10/11 build 号 < 65535，均在 16 位内
- **`nt_version` 必须在 Cargo.toml 中用 `[target.'cfg(windows)'.dependencies]` 条件引入**：crate lib.rs 含 `#[cfg(not(windows))] compile_error!`，非 Windows 编译硬失败。命令函数同样需 `#[cfg(windows)]` 守卫
- **F2 的 `dispose()` 自动回退 DOM 渲染器**：xterm.js `WebglAddon.dispose()` 内部调用 `_createRenderer()` 创建 DomRenderer 并 `renderService.setRenderer()`——白屏不会发生
- **F3 默认值**：如果 `invoke('get_windows_build_number')` 失败（非 Windows 平台），fallback 到 `21376`（ConPTY 阈值），保证终端仍可工作
- **F4 脚本的 `$?` 恢复必须是 prompt 函数最后有效语句**：`1+1`（设 $?=$true）/ `Write-Error '' -ErrorAction Ignore`（设 $?=$false）。Starship 验证模式
- **`-EncodedCommand` 编码格式**：UTF-16LE（无 BOM）+ 标准 Base64（`general_purpose::STANDARD`）
- **F1+F2+F3 测试更新**：workspace.test.tsx 需适配新 watermark（含按钮），terminal.test.tsx 需适配新 useXterm 签名
- **L1 测试中的 `test_shell_integration_script_embedded`**：F4 会改脚本内容，但断言 `!script.is_empty()` + `script.contains("OSC")` 仍成立——无需改测试
