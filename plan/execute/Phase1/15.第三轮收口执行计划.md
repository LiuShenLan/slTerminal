## 二、修复方案

### 2.1 G2 — Watermark/RHA 终端创建不可用（D2）

**根因**：Dockview `addPanel()` 后 CSS 布局通过 rAF 异步提交。Watermark 空态下新面板容器首次渲染尺寸为 0×0 → `fitAddon.fit()` 中 `proposeDimensions()` 返回 `undefined` → 静默放弃。ResizeObserver 在 WebView2 中可能触发一次 {0,0} 后永不再触发（CSSWG #7808）。

**修复（5 层守卫）**：

| 层 | 内容 | 来源 |
|----|------|------|
| 1 | `new Terminal({ cols: 80, rows: 24 })` 构造函数安全网 | Daintree |
| 2 | ResizeObserver 注册与 Terminal 创建解耦为独立 useEffect（解决 StrictMode 守卫截断后 RO 无法重建的问题） | RO agent 发现 |
| 3 | `canFit()` 五条件显式守卫替代 try-catch（export 为纯函数，可测试） | OpenHands PR #12095 |
| 4 | ResizeObserver 门控：首次同步 fit，后续 rAF 去重 | Daintree |
| 5 | setTimeout 回退 [0, 50, 200]ms × 3（WebView2 RO {0,0} 静默失败回退） | 自定义 |

**裁剪项**（与审计 §3.1.1 相比）：

| 裁剪 | 理由 |
|------|------|
| `contain: strict` CSS | 解决多终端（10-30个）级联重排；单终端无此问题；Daintree 自己在 grid cell 禁用（#4713）；会破坏 `position: fixed` |
| `visibilitychange` + `pendingFitRef` | WebView2 `visibilityState` 最小化时保持 `"visible"`（MS 官方文档）；事件永不触发；Tauri 修复已回滚（#10592） |
| `MIN_CONTAINER_SIZE = 50` | `offsetWidth > 0` 已精确拦截 display:none；FitAddon 内置 `MINIMUM_COLS=2 / MINIMUM_ROWS=1` |
| `useLayoutEffect` | 保持 `useEffect`；当前守卫模式正确；改 `useLayoutEffect` 不会修复已知 bug |
| setTimeout 5 次 → 3 次 | VS Code 用常量 50ms×5；WebView2Feedback #2776 ~12ms 修正窗口，3 次已覆盖 |

**`canFit()` 条件**：

```typescript
// export 为纯函数，src/__tests__/canFit.test.ts 可独立测试
export function canFit(
    terminal: Terminal | null | undefined,
    fitAddon: FitAddon | null | undefined,
    containerEl: HTMLElement | null | undefined,
    isDisposedRef: { current: boolean }
): boolean {
    if (!terminal || !fitAddon) return false;    // 条件1: 实例存在
    if (!containerEl) return false;               // 条件2: 容器存在
    if (containerEl.offsetWidth === 0 ||           // 条件3: 非 display:none
        containerEl.offsetHeight === 0) return false;
    if (!terminal.element) return false;           // 条件4: open() 已调用
    if (isDisposedRef.current) return false;       // 条件5: 未销毁（手动 useRef 维护）
    return true;
}
```

注：xterm.js Terminal 类**没有公开的 `isDisposed` 属性**——需 `useRef` 手动维护。FitAddon 的 MINIMUM_COLS=2 / MINIMUM_ROWS=1 / isNaN 守卫 / cell.width===0 守卫已内置，无需在 `canFit` 中重复。

**改动文件**：`src/panels/terminal/useXterm.ts` ~35 行，`src/panels/terminal/TerminalPanel.tsx` ~6 行

### 2.2 F3 — buildNumber 时序 bug + 掩码改进（D6）

#### Bug 1：`windowsPty` 从未生效

**根因**：`useXterm.ts` 的 useEffect 首次执行时 `windowsBuildNumber === undefined`（IPC 未返回）→ 跳过 `windowsPty` 设置。IPC 异步返回后 `windowsBuildNumber` 变为真实值 → useEffect 因依赖变化重跑 → `terminalRef.current` 守卫（L97）拦截，永远不到达 `windowsPty` 设置代码。

xterm.js `_handleWindowsPtyOptionChange` **在 Terminal 创建后设置 `options.windowsPty = {...}`（新对象）仍会触发 handler**——真正问题是守卫阻止了设置代码的执行。

**修复**：独立 useEffect 监听 `windowsBuildNumber`，在 Terminal 已创建后赋值：

```typescript
// 独立 useEffect——不参与 Terminal 创建的"仅一次"守卫
useEffect(() => {
    const term = terminalRef.current;
    if (term && windowsBuildNumber !== undefined) {
        term.options.windowsPty = {
            backend: "conpty",
            buildNumber: windowsBuildNumber,
        };
    }
}, [windowsBuildNumber]);
```

**QA 验证**（`TerminalPanel.tsx`）：

```typescript
// 改前
pty.getWindowsBuildNumber().then((bn) => setBuildNumber(bn));
// 改后
pty.getWindowsBuildNumber().then((bn) => {
    console.log('[slTerminal] Windows build:', bn);
    setBuildNumber(bn);
});
```

验证步骤：DevTools Console 看到 `[slTerminal] Windows build: 26100` → PowerShell `(Get-ItemProperty "HKLM:\SOFTWARE\Microsoft\Windows NT\CurrentVersion").CurrentBuild` 交叉验证一致。

#### Bug 2：掩码语义改进（🟡）

`RtlGetNtVersionNumbers` 返回 raw build = `0xF0006658`。高 4 位（bit 28-31）= `0xF` = 零售版标记。`& 0xFFFF` 提取低 16 位——这是 Cygwin/Go/ReactOS/Wine 四家共同的事实标准，当前所有 Windows build < 65535，功能正确。

改为 `& 0x0FFFFFFF`（清除高 4 位标志，保留低 28 位）——语义更清晰表达"清除构建类型标志"的意图，面向未来。

```rust
// lib.rs:27
// 改前
Ok(build & 0xFFFF)
// 改后
Ok(build & 0x0FFFFFFF)  // 清除高 4 位构建类型标志（0xF=零售版），低 28 位为实际 build 号
```

**改动文件**：`useXterm.ts` +7 行，`TerminalPanel.tsx` 1 行改，`lib.rs` 1 行改

### 2.3 D3 — 编辑器语法高亮（D3）

**根因**：`useCodeMirror.ts` 的 extensions 配置含 `basicSetup`（18 项扩展）和 `oneDark`（主题+高亮样式），但**不含任何 Lezer grammar**。`basicSetup` 中的 `syntaxHighlighting(defaultHighlightStyle, { fallback: true })` 仅提供高亮 CSS 类框架——不产生任何 token。

CodeMirror 6 动态切换语言的唯一方案：`Compartment + reconfigure`。`view.dispatch({ effects: langCompartment.reconfigure(newLang()) })` **不丢失文档状态**。

**修复**：

1. 在 `useCodeMirror.ts` 中新增 `getLanguageExtension(filename)` 纯函数（export 可测试）
2. 新建 `Compartment` 实例（`useMemo` 稳定引用）
3. 扩展名→语言包映射表（8 种语言）：
   - `.js/.ts/.tsx/.jsx/.mjs/.cjs` → `javascript()`
   - `.py/.pyw` → `python()`
   - `.rs` → `rust()`
   - `.json/.jsonc` → `json()`
   - `.html/.htm` → `html()`
   - `.css/.scss/.less` → `css()`
   - `.md/.markdown` → `markdown()`
   - `.xml/.svg` → `xml()`
   - 无扩展名/未知 → `javascript()`（默认）
4. G3 另存为后更新 `filePathRef.current` → 在 `filePath` 变化时调用 `reconfigure`

注：8 个 `@codemirror/lang-*` 包已安装在 `package.json` 中——此前为死依赖，修复后全部被引用。

**改动文件**：`useCodeMirror.ts` ~15 行

### 2.4 D4 — dialog 架构违规（D4）

**根因**：`useCodeMirror.ts:14` 的 `import { save } from "@tauri-apps/plugin-dialog"` 违反架构约束 #1（"invoke 只允许出现在 `src/ipc/`"）。ESLint `no-restricted-imports` 规则仅限 `@tauri-apps/api/core` 的 `invoke`，未覆盖 `@tauri-apps/plugin-*`。

`@tauri-apps/plugin-dialog` 的 `save()` 本身就是 `invoke()` 的薄包装——走相同 IPC 通道，零额外安全风险。但约束 #1 的**字面要求**是集中式 IPC 审计点。

**修复**：

1. 新建 `src/ipc/dialog.ts`——薄 re-export（3 行）：
   ```typescript
   // @tauri-apps/plugin-dialog 的 save() 本身就是 invoke() 包装器——薄 re-export 零开销
   export { save } from '@tauri-apps/plugin-dialog';
   ```

2. ESLint `patterns` 全局封堵（`eslint.config.js` +6 行）：
   ```javascript
   patterns: [{
     group: ['@tauri-apps/plugin-*', '@tauri-apps/plugin-*/**'],
     message: '所有 Tauri 插件 API 请通过 src/ipc/ 封装。',
   }]
   ```

3. `useCodeMirror.ts` import 路径改为 `import { save } from "../../ipc/dialog";`（1 行改）

注：ESLint `no-restricted-imports` 的 `patterns` 使用 npm `ignore` 包（gitignore 规范），`@tauri-apps/plugin-*` glob 完全合法。Vite/Rollup 对 ESM 做 AST 级 tree-shaking，薄 re-export 零 bundle 体积影响。

**改动文件**：`src/ipc/dialog.ts` 3 行（新文件），`eslint.config.js` +6 行，`useCodeMirror.ts` 1 行改

#### 4.0.3 `/goal` 与 Dynamic Workflow 的关系

```
/goal 启动
  │
  ├─ Turn 1: Agent 检测基线（cargo test + npm test 贴输出）→ 基线全绿
  │   └─ 调用 Workflow 工具 → 3 agent 并行修改代码
  │       └─ ★ 关键：workflow 回归阶段 Agent 把 cargo test/npm test/tsc/eslint/clippy
  │          输出全部贴回主会话 → Evaluator 在下一轮可见这些输出
  ├─ Turn 2: Evaluator 读主会话对话 → 检查 8 条条件
  │   ├─ 全部满足 → /goal 完成 ✓
  │   └─ 有未满足项 → Agent 修复 → 贴修复后命令输出
  ├─ Turn 3+: 同上循环
  └─ Turn 30: 仍未满足 → 停止并逐项汇报缺口
```

**Evaluator 可见性规则**：
- ✅ **可见**：Agent 在主会话中 `Bash`/`PowerShell` 工具的输出、Agent 的文本回复
- ❌ **不可见**：workflow 子 agent 内部对话、子 agent 互发消息、未贴回主会话的命令输出
- ⇒ **workflow 回归阶段必须把验证命令输出贴回主会话**，否则 Evaluator 只能看到 Turn 1 的基线旧输出

**为什么不是 5 个独立的 `/goal`**：`/goal` 是"目标驱动循环"，Evaluator 只看终态条件——不管中间经过几个 Turn。§4.1 的 Turn 标注是**概念阶段**（基线→实现→回归→修→报告），是 Agent 的工作计划，不是 5 条 `/goal` 命令。

## 七、与审计文档的差异说明

以下审计文档中的主张经 6 轮网络检索后修正：

| 审计主张 | 检索发现 | 修正 |
|---------|---------|------|
| `contain: strict` CSS 纳入 Layer | 多终端级联重排方案，单终端不需要 | 裁剪 |
| `visibilitychange` + `pendingFitRef` | WebView2 不触发，Tauri 修复已回滚 | 裁剪 |
| `MIN_CONTAINER_SIZE = 50` | `offsetWidth > 0` + FitAddon 内置 MINIMUM_COLS=2 已覆盖 | 裁剪 |
| setTimeout [0,50,100,200,500] 来自 VS Code | VS Code 用常量 50ms×5 | 序列自定义，降为 3 次 |
| WebGL "双倍纹理图集" | 图集缓存键不含 cols/rows | 描述降级，方案不变 |
| `terminal.isDisposed` 存在 | Terminal 类无公开 `isDisposed` 属性 | 改为 `useRef` 手动维护 |
| `& 0xFFFF` 是 bug | Cygwin/Go/ReactOS/Wine 四家事实标准 | 降级为 🟡 语义改进 |
