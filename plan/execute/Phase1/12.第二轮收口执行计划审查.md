### G1b — pty_kill async + spawn_blocking ✅⚠️

| 验证点 | 来源 | 结论 |
|--------|------|------|
| 当前代码在持有 RwLock 写锁时执行 `handle.join()` + `session.drop()` | 本地源码 `spawn.rs:178-197` | ✅ 根因诊断正确 |
| portable-pty 0.9.0 `PseudoCon::drop` 在 pre-Win11 24H2 上调用阻塞版 `ClosePseudoConsole` | portable-pty 0.9.0 用 `std::mem::forget(pair.master)` 泄漏句柄绕过（commit `7e50c4db`） | ✅ 阻塞机制确认 |
| `State<'_, AppState>` 不 move 进 spawn_blocking——仅 extract 后的 owned `session` 数据进入 | Rust 借用规则 | ✅ 内存安全 |
| Tauri 2 `generate_handler!` 原生支持 `async fn` | Tauri 2 文档 | ✅ 无需宏改动 |

⚠️ **风险**：`session.drop()` 在 `spawn_blocking` 中触发 `ClosePseudoConsole`。

> 🔍 **核查结论：⚠️ 审查风险描述严重度被低估。`ClosePseudoConsole` 阻塞是 INFINITE（永久），非"数秒延迟"。**
>
> **证据链**（Microsoft 文档 + 源码逆向 + 多项目实锤）：
> 1. **Windows 版本差异**（Microsoft Learn 官方文档）：pre-Win11 24H2 (build < 26100) 上 `ClosePseudoConsole` 调用 `WaitForSingleObjectEx(INFINITE)`——**无限期阻塞**等待 conhost 退出。Win11 24H2+ 改为立即返回。审查文档写的 "Win10 build < 22621" 应为 "build < 26100"——22621 (Win11 22H2) 同样阻塞。
> 2. **阻塞时长是 INFINITE**（非数秒）：如果输出管道未排空且无人读取，conhost 的 `RenderThread` 持有 ticket lock 阻塞在 `WriteFile` 上 → conhost 永远无法退出 → `ClosePseudoConsole` 永久挂起。来源：microsoft/terminal Discussion #17716 (lhecker 确认)、Alacritty conpty.rs 源码注释 "This will cause a deadlock if the conout pipe has already been dropped"。
> 3. **`spawn_blocking` 线程池上限 512**（Tokio 源码 `max_blocking_threads: 512`）。实际场景下 512 个并发 pty_kill 不会发生——但风险存在：如果 N 个 terminal 同时关闭且都触发 ClosePseudoConsole 挂起，N 个 blocking 线程永久占用。
> 4. **wezterm PR #5977 未合并**：截至 2026-06-19 仍为 **Open**（GitHub API 确认：`state: "open"`, `merged: false`, `merged_at: null`，自 2024-08-16 创建，最后活动 2026-06-06）。portable-pty 0.9.0 不含任何 ClosePseudoConsole 修复——crates.io 最新仍是 0.9.0（2025-02-11，仅替换依赖 `serial`→`serial2`）。PR #5977 的 CloseHandle+drain 方案尚未合并到 wezterm main，也未发布到 crates.io。
>    > 🔍 **本核查轮追加验证（GitHub API）**：PR #5977 确认为 Open。commit `7e50c4db68`（Wez Furlong，2019-07-17）是 wezterm 应用层针对 daemonized WSL 场景的 `std::mem::forget(pair.master)` workaround——**不是** portable-pty crate 的通用修复，不在 `pseudocon.rs` 中。
> 5. **已知受影响的真实项目**：Turborepo (PR #11816)、wezterm (Issue #5882)、VS Code (terminal #1810)、node-pty (PR #415)、Alacritty——全部证实 pre-24H2 上 ClosePseudoConsole 死锁。
>
> **修正风险评估**：`spawn_blocking` 将 `ClosePseudoConsole` 从 IPC worker 线程移到阻塞线程池——IPC 不阻塞（目标达成）。但若用户在短时间内连续关闭多个 terminal（如"全部关闭"），每个 terminal 各占一个 blocking 线程——在 slTerminal 的终端数量级（~10 个）下远未触及 512 上限。**实际风险：低。** 长期建议：portable-pty 升级到包含 PR #5977 修复的版本后移除关注。
>
> **审查文档一处事实性错误**：阈值 build 是 26100 (24H2)，非 22621 (22H2)。22621 的 `ClosePseudoConsole` 仍阻塞。

**结论**：方案正确。建议在注意事项中补充 portable-pty 0.9.0 的 `ClosePseudoConsole` 阻塞版本说明。

⚠️ **风险—事件循环死锁**：审查文档引用 Tauri issues #7673、#10586、#11212 论证 dialog 事件循环死锁风险。

> 🔍 **核查结论（两轮交叉核验 + GitHub API 确证）**：
> 1. **#7673 — ❌ 误引**：Tauri v2.0.0-alpha.11 IPC 参数名冲突 bug，alpha.12 已修复。与 v2.11.x 无关。
> 2. **#10586 — ❌ 跨版本误引**：Tauri **v1**.7.1 macOS 偶发冻结。v2 架构完全独立（v1 内建 API → v2 独立插件 + async IPC）。
> 3. **#11212 — ❌ 归类错误**：属于 `tauri-plugin-shell`（`shell.open()` 同步阻塞），非 dialog。已修复。
> 4. **#2268 — ⚠️ 真实风险（审查遗漏）**：仅 `import { open }` 即导致条件渲染列表内 `<input>` 失焦/IME 无法输入。**至今 OPEN**（自 2024-12，v2.3–2.7 均无修复，无维护者回复）。环境：React 18（非 19），未提 Vite。slTerminal 中 dialog 仅在 `useCodeMirror.ts` 使用——CodeMirror 不含 `<input>`，实际影响极小。
> 5. **`dialog.save()` 为 async IPC**：JS 端 Promise，Rust 端 `pub async fn`。对话框期间 JS 事件循环继续运行——非"模态阻塞"。

**结论**：方案正确。添加事件循环风险提示即可。

## 三、遗漏项

### 3.1 G3 缺少 TypeScript 编译验收

G3 引入 `@tauri-apps/plugin-dialog` npm 包后，`useCodeMirror.ts` 中的 `import { save }` 需要 TypeScript 类型声明。虽然 `@tauri-apps/plugin-dialog` 自带 `.d.ts`，但计划验收中未包含 `npx tsc --noEmit` 检查。

**建议**：G3 验收项增加 `npx tsc --noEmit` 退出 0。

### 3.2 缺少 `tauri-plugin-dialog` npm 包安装步骤

G3 操作第①步写 `npm install @tauri-apps/plugin-dialog@^2`，但 agent 核验确认当前代码库**尚未集成此插件**。操作步骤正确但需注意——安装后需在 workspace.test.tsx 的 mockIPC 中增加 `plugin:dialog|save` 的 mock 处理（如果测试路径触发 `useCodeMirror`）。
