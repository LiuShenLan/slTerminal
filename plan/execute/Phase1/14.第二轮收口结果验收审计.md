## 3. 人工验收失败项分析

### 3.1 G2 未修复：Watermark/RHA 终端创建后不可用（🔴 阻断）

**现象**（报告 §2.1）：

| # | 创建路径 | 结果 |
|---|---------|------|
| #4 | Watermark"新建终端" | 终端面板创建成功，但只有特别小的输入框，没填充屏幕，无 PowerShell 提示符 |
| #6 | 工具栏"+"按钮 | 终端面板创建成功，但终端不可用 |
| #7 | 右键菜单"新建终端" | ✅ 通过 |

**根因分析**（3 轮共 7 路 agent 网络检索 + Dockview/xterm.js/Daintree 源码确证）：

执行计划 §7 声称 "G2 无需独立修复——G2 是 G1a 的级联效应，随 G1a 修复自然恢复"。**此判断不成立。** G1a 修的是 `onReady` 在 React StrictMode 下双发导致 Dockview 内部状态异常。G2 是**不同的根因**——容器布局时序问题：

1. **Dockview CSS 布局异步提交**（dockview-core 源码确证）：`addPanel()` 后 Dockview 内部通过 `requestAnimationFrame` 异步提交 CSS 布局。在 Watermark 空态下，新面板的容器在首次渲染时尺寸为 **0×0**。来源：Dockview #318（浮动面板 `getBoundingClientRect()` 返回 0）、#640（面板尺寸约束 bug）、#866（StrictMode 下 SplitviewReact 渲染两次）。

2. **xterm.js FitAddon 在 0×0 容器中静默失败**：`fit()` 调用 `proposeDimensions()` → 容器尺寸为 0 → `css.cell.width = 0` → 返回 `undefined` → `fit()` 静默放弃。xterm.js issue #4578（维护者 jerch 定性为"已知限制"）、#4853（FitAddon 错误尺寸）。

3. **ResizeObserver 未恢复**：代码中已有 `ResizeObserver`（`useXterm.ts:150-161`），理论上容器尺寸变化后应触发 `fitAddon.fit()`。但 WebView2 中 ResizeObserver 触发时序与标准 Chrome 不同——canopyide/canopy #4913 确证：Windows 上 `.xterm-screen` 元素存在但永远不可见，Linux 修复（PR #4889）后 Windows 仍失败。Monaco #4333 确认"ResizeObserver loop limit exceeded"**仅限 WebView2**，不发生在 Chrome。

4. **右键菜单场景正常**：右键菜单触发时 Dockview 已有面板且布局稳定 → 新面板容器获得 `existingW × existingH` → FitAddon 正常工作。

**Chromium 渲染管道规范确证**（Chromium Blink 源码 + WPT）：

```
Tasks → Microtasks → rAF callbacks → ResizeObserver → Style → Layout → Paint
```

- **rAF 在布局之前触发**（不是之后！）——单次 rAF 不保证布局完成
- **ResizeObserver 在布局之后触发**——是检测布局就绪的正确信号
- **双 rAF 保证布局至少运行过一次**（Mozilla 源码注释 + WPT #16123），但 WebView2 调整大小时可能在两帧间插入新布局失效

**Daintree（原 Canopy）生产级修复**（PR #4935，2026-04-03 合并）：

Daintree 经过 10+ issue 迭代（#602/#847/#4889/#4913/#4429/#1177/#1806/#2221），打磨出当前生产代码的 **5 层连锁防御**。核心文件 `src/components/Terminal/XtermAdapter.tsx`（705 行），关键常量 `MIN_CONTAINER_SIZE = 50`。

但 slTerminal 场景比 Daintree 简单——单窗口、`renderer:'always'`、无 visibilitychange 切换——不需要 Daintree 的全部层级。

### 3.1.1 G2 修复方案（grill 决策 D2）

**方案：ResizeObserver 门控 + setTimeout 重试回退 + 构造函数初始尺寸安全网（6 层守卫）**

```
Layer 1: new Terminal({ cols: 80, rows: 24 })     ← 构造函数安全网
         （优于 term.resize(80,24)——效果相同但避免 WebGL 双倍纹理图集创建，
         来源：xterm.js 源码 Terminal.ts:147 + WebGL addon 纹理管线分析）
Layer 2: useLayoutEffect（非 useEffect）          ← 防 React StrictMode 双重挂载
         （来源：use-resize-observer #90/#91/#92——useEffect cleanup 后 didUnmount
         ref 未重置导致 remount 后 RO 回调被阻塞）
Layer 3: ResizeObserver → contentRect >= 50px → fitAddon.fit() → disconnect
         （一次性门控，来源：Daintree PR #4935 + OpenHands PR #12095）
Layer 4: setTimeout 重试 [0, 50, 100, 200, 500]ms × 5
         （回退——WebView2 中 RO 可能触发一次{0,0}后永不再触发，
         CSSWG #7808 确认此规范漏洞仍 Open）
Layer 5: fit() 守卫：if (cols < 2 || rows < 1) return
         （防 PTY 截断——窗口最小化时 WebView 客户区 0×0 → fit 计算 cols=2/rows=1
         → PTY 历史数据不可逆丢失。来源：godly-terminal #154 +
         Microsoft Terminal commit c5d417f）
Layer 6: isDisposed 守卫 → 防 StrictMode remount 后回调操作已释放资源
```

**关键修正**（与前一轮初审建议相比）：
- ~~双 rAF~~ → ResizeObserver 门控（rAF 在布局前触发，不是后）
- ~~term.resize(80,24)~~ → `new Terminal({ cols, rows })`（避免 WebGL 双倍图集）
- ~~仅 ResizeObserver~~ → RO + setTimeout 重试回退（CSSWG #7808：RO 可能触发一次{0,0}后永不再触发）
- 新增 50px 最小容器尺寸阈值（来自 Daintree 生产实践）
- 新增 PTY 截断防护（最小化场景）
- 新增 isDisposed 守卫（StrictMode 双重挂载）

**改动量**：`useXterm.ts` ~45 行，`TerminalPanel.tsx` ~5 行（构造函数 cols/rows 传参）

**证据链**：
| 层级 | 来源 | 关键证据 |
|------|------|---------|
| Chromium 渲染管道 | Blink 源码 `LocalFrameView::NotifyResizeObservers()` | rAF before layout, RO after layout |
| CSSWG #7808 | CSSWG 规范仓库 | RO may fire {0,0} once then never again |
| Daintree PR #4935 | `.lessons/4935.md` | visibilityState gate + useLayoutEffect fix |
| xterm.js #4578 | 维护者 jerch | resize+fit is workaround, not guaranteed |
| godly-terminal #154 | GitHub issue | window minimize → PTY data loss |

| WebView2Feedback #2776 | Microsoft Edge feedback | initial size wrong, ~12ms correction window |

### 3.2 编辑器无语法高亮（🟡 中等）

**现象**（报告 §2.3 #17）："文本正常显示，没有语法高亮"

**根因确证**（CodeMirror 官方文档 + 作者 Marijn 确认 + GitHub 讨论 #5284/#6003/#6805/#8056）：

`useCodeMirror.ts` 的 extensions 配置为：

```typescript
extensions: [basicSetup, oneDark, saveKeymap],
```

**`basicSetup` 不含语言特定的语法高亮。** 这是 CodeMirror 6 的刻意设计——`basicSetup` 中的 `syntaxHighlighting(defaultHighlightStyle)` 仅提供高亮 CSS 类框架（token→颜色映射），**不产生任何 token**。Token 必须由语言包（Lezer grammar）产生。

> CodeMirror 作者 Marijn Haverbeke："basicSetup does not include a language extension. You need to add e.g. `javascript()` from `@codemirror/lang-javascript`."

**最小可用组合**：`extensions: [basicSetup, javascript(), oneDark, saveKeymap]`

**死依赖**：`package.json` 中安装了 8 个 `@codemirror/lang-*` 包（javascript/python/rust/json/html/css/markdown/xml），但**全部未被 `useCodeMirror.ts` 导入或使用**——纯死依赖。

### 3.3 `@tauri-apps/plugin-dialog` 架构违规（🟡 中等）

**现象**：`useCodeMirror.ts:14`：

```typescript
import { save } from "@tauri-apps/plugin-dialog";
```

违反架构约束 #1："前端绝不直接碰 OS/文件/进程：`invoke` 只允许出现在 `src/ipc/`；其它文件只调用 `ipc/` 暴露的领域函数。"

**深度分析**（4 路 agent 并行检索——Tauri 插件 IPC 机制 + 14 个生产项目架构 + ACL 安全模型 + ESLint 边界规则）：

**技术事实（plugin-dialog 源码审查）**：

`@tauri-apps/plugin-dialog` 的 `save()` **本身就是 `invoke()` 的薄包装**：

```js
// node_modules/@tauri-apps/plugin-dialog/dist-js/index.js
import { invoke } from '@tauri-apps/api/core';
async function save(options = {}) {
    if (typeof options === 'object') Object.freeze(options);
    return await invoke('plugin:dialog|save', { options });
}
```

- 与自定义 `#[tauri::command]` 走**完全相同的 IPC 通道**（JSON 序列化 → IPC 协议 → Rust dispatch → ACL 检查）
- 插件命令的 ACL 检查**比自定义命令更严格**（`plugin_command.is_some()` 强制触发，本地 origin 无 AppManifest 时自定义命令可绕过 ACL，插件不可）
- JS 端零安全边界——真正安全在 Rust 后端 ACL
- Tauri 官方文档：插件 API 包就是让开发者直接导入的

**业界实践（14 个 Tauri 2 生产项目审查）**：

| 模式 | 占比 | 代表项目 |
|------|------|---------|
| 插件也走 IPC 层封装 | ~60% | Spacedrive (34k★)、Chatbox (31k★)、Clash Verge Rev、HuLa、Kubeli |
| 插件直接导入 | ~10% | Loom、lencx/ChatGPT（已归档） |
| 代码生成替代手动封装 | ~10% | Spacedrive (Specta) |
| 薄工具函数包装 | ~20% | Re-prod (`openFolder()` 包装 `open()`) |

**矛盾**：技术上插件函数就是 invoke 包装器——再包一层是空函数转发，不增加安全价值。但 ~60% 生产项目仍然包装——理由是一致性、可测试性、集中审计。

### 3.6 buildNumber 验证方法缺失 + **F3 时序 bug** + **掩码 bug**（🔴 阻断）

**原现象**（报告 §2.8 #35）：用户注记"这个应该怎么执行？"——验收清单未提供可操作的验证方法。

**Grill 深度检索发现的额外 bug**：

#### Bug 1：F3 `windowsPty.buildNumber` 从未生效（xterm.js 源码分析）

`useXterm.ts` 的执行顺序存在时序缺陷：

```
1. 首次 render: windowsBuildNumber = undefined
   → useEffect 触发 → terminalRef.current = null → 创建 Terminal
   → windowsBuildNumber === undefined → 跳过第 103-108 行 windowsPty 设置
   → terminalRef.current = term（已创建，无 windowsPty 配置）

2. IPC 异步返回: windowsBuildNumber = 26200
   → useEffect 再次触发（依赖数组含 windowsBuildNumber）
   → L97: if (terminalRef.current) return; ← 直接返回！
   → Terminal 的 windowsPty 永远没被设置
```

**影响**：xterm.js 始终用默认行为，ConPTY reflow/换行启发式未按 OS build 配置。**F3 功能名存实亡。**

xterm.js v6 内部决策逻辑（`CoreTerminal._handleWindowsPtyOptionChange()`）：

| 条件 | 换行启发式 | Reflow |
|------|-----------|--------|
| `backend='conpty'` + `buildNumber >= 21376` | 关闭 | 开启 |
| `backend='conpty'` + `buildNumber < 21376` | 开启 | 关闭 |
| 未设置 `windowsPty` | 关闭 | 开启（默认） |

#### Bug 2：buildNumber 掩码过窄（nt_version 源码分析 + 实机测试）

Agent 在实机上跑 `RtlGetNtVersionNumbers` 获取原始值：

| 数据源 | 返回值 |
|--------|--------|
| `RtlGetNtVersionNumbers` raw build | `0xF0006658` |
| `build & 0xFFFF`（当前） | `26200`（碰巧正确，build < 65535） |
| `build & 0x0FFFFFFF`（正确掩码） | `26200` |

高 4 位（bit 28–31）= `0xF` = 零售版标记。真实 build 在低 28 位。当前 `& 0xFFFF` 在 build > 65535 时会截断。`& 0x0FFFFFFF` 是 Cygwin/MSYS2/ReactOS/Wine 公认的惯例掩码。

### 3.6.1 buildNumber 修复方案（grill 决策 D6）

**修复 1**：`useXterm.ts` 加独立 `useEffect` 处理异步更新的 `windowsBuildNumber`

```typescript
// 新建独立 useEffect：buildNumber 异步返回后更新已有 Terminal 的 windowsPty
useEffect(() => {
  if (windowsBuildNumber !== undefined && terminalRef.current) {
    terminalRef.current.options.windowsPty = {
      backend: "conpty",
      buildNumber: windowsBuildNumber,
    };
  }
```

## 10. 附录：本轮审计方法学记录

### A.1 本轮新增识别

| 模式 | 描述 |
|------|------|
| **"随 X 自动恢复"假设的危险** | 执行计划声称 G2 随 G1a 自动恢复，实际根因不同。修复项之间的"级联效应"假设必须经代码路径追踪验证——不能仅凭直觉推断 |

| **插件 import 绕开架构约束** | ESLint 规则仅限 `@tauri-apps/api/core` 的 `invoke`，`@tauri-apps/plugin-dialog` 的 `save()` 同样是 IPC 调用但未被覆盖 |

| **掩码过窄的潜在 bug** | `& 0xFFFF` 在 build < 65535 时碰巧正确——低概率 bug 在代码审查中容易被忽略，需查 API 实际返回值结构（`RtlGetNtVersionNumbers` 高 4 位是 `0xF` 标记） |
