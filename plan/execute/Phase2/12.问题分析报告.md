## 问题 1：TerminalRegistry 无自动化测试覆盖

**描述**：H6-A1 新建 `TerminalRegistry.ts` 模块（~40 行），暴露 `register/get/remove/has` 四个方法。该模块是终端跨页面复用的核心基础设施，但无任何单元测试。`npm test`（49 tests）中无 `TerminalRegistry.test.ts`。H6-A2（useXterm 查 Registry 再 spawn）和 H6-A3（cleanup 中 isLayoutSwitching 分支）的行为变更同样无测试覆盖——人工验证确认 H6 仍未通过（"新拉起了一个新的终端"），但自动化无从知晓。

### 根因

Phase 2 执行计划（`10.修复执行计划.md`）的 H6-A1（创建 TerminalRegistry 模块）、H6-A2（useXterm 查 Registry 再 spawn）、H6-A3（cleanup 中 isLayoutSwitching 分支）三个修复点均未包含对应的自动化测试任务。执行时只创建了 `TerminalRegistry.ts` 和修改了 `useXterm.ts`，未同步创建测试文件。

### 修复方式

新建 `src/__tests__/TerminalRegistry.test.ts`（7 条：register/get/remove/has 往返、幂等覆盖、_clear 清空）和 `src/__tests__/terminal-lifecycle.test.ts`（6 条：通过 render + unmount + `isLayoutSwitching` 标志模拟 H6 三条路径）。`npm test` 从 49→62（+13），T2 门自动覆盖变更行为。

> 技术要点：Terminal mock 必须用 `vi.fn(function() {...})`（非箭头函数），否则 `new Terminal()` 报 "is not a constructor"。`useFakeTimers()` + `runAllTimersAsync()` 控制 rAF 轮询+异步 spawn 时序。

### 已经排除的原因

- **jsdom 无法 mock `new` 构造器** → 排除。`vi.fn(function() {...})`（普通函数）可正常被 `new` 调用。
- **rAF 轮询 + 异步 spawn 无法在测试中控制** → 排除。`vi.useFakeTimers()` + `vi.runAllTimersAsync()` 可完整控制时序。
- **需要真实 WebGL context** → 排除。jsdom 中 `canvas.getContext('webgl2')` 返回 null，useXterm 内 `detectWebgl()` 返回 false，自动回退 DOM 渲染器，不影响测试逻辑。
- **需要完整 Terminal/FitAddon 接口实现** → 排除。`as unknown as Terminal` 最小 stub 模式（参照 `canFit.test.ts`）足够满足测试需求。

## 问题 2：onCloseRequested 关闭钩子无自动化测试覆盖

**描述**：S4-D4 在 `App.tsx` 新增 `appWindow.onCloseRequested` 异步关闭钩子（~35 行），包含 layout flush → saveAllProjects → localStorage → destroy 四步序列。该钩子无任何测试覆盖。人工验证发现"点击客户端的关闭按钮，无法关闭客户端"——`onCloseRequested` 中 `event.preventDefault()` + 异步 `destroy()` 的组合行为未经验证，可能因 `destroy()` 调用失败或 Tauri API 行为差异导致窗口无法关闭。

### 根因

S4-D4 修复在 `App.tsx` 新增 `onCloseRequested` 异步关闭钩子，包含 `preventDefault` → flush layout → `saveAllProjects` → `localStorage` → `destroy` 四步序列。执行计划中该修复项未包含对应测试任务，约 35 行关键逻辑（含错误吞没、try/catch 边界、`destroy()` 是否在 catch 外）无任何自动化覆盖。人工验证发现"点击关闭按钮无法关闭客户端"，但自动化无从定位。

### 修复方式

新建 `src/__tests__/close-handler.test.ts`（4 条：正常关闭四步序列、saveAllProjects 异常不阻塞 destroy、无 activePageId 跳过 flush+localStorage、无 `__dockviewApi` 跳过 flush）。通过 `vi.mock` 替代 `@tauri-apps/api/window` + stores + Workspace，`render(<App />)` 后捕获 `onCloseRequested` 回调，模拟关闭事件验证四步序列及错误隔离。`npm test` 从 62→66（+4），T2 门自动覆盖。

> 技术要点：`activePageId` 存入 hoisted mutable 变量（`mocks.mockActivePageId`），避免 `mockReturnValue` 状态泄露到后续测试。`localStorage` 用内存 stub 替代（Node 22 jsdom 中 `localStorage` 需 `--localstorage-file` 标志，不可用）。

### 已经排除的原因

- **无法在 jsdom 中测试 Tauri window API** → 排除。`vi.mock("@tauri-apps/api/window")` 可完整替代 `getCurrentWindow`/`onCloseRequested`/`destroy`。
- **App.tsx 渲染链涉及 Dockview，无法测试** → 排除。`vi.mock("../workspace")` 替代 Workspace 为最小 div，不触发 Dockview 初始化。
- **async handler 注册时机不可控** → 排除。`onCloseRequested` 在 useEffect 中同步注册，`render()` 后 handler 立即可用（`waitFor` 轮询）。
- **`event.preventDefault()` 的 Tauri 行为差异** → 排除。测试覆盖的是 handler 内部逻辑正确性（preventDefault 被调用 + destroy 在 catch 外确保执行），而非 Tauri 原生关闭行为——后者属 E2E 范畴。
