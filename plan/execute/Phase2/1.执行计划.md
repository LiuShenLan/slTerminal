## 1. 编排总览

### 1.1 为什么这么编排

Phase 2 是**混合型**任务——后端 git 命令与前端 UI 可并行开发，但前端 stores/ipc 层依赖后端命令完成：

```
后端 git 命令（git_is_repo, git_worktree_*）
  → 前端 IPC 层（ipc/git.ts）
    → stores（projects.ts, layout.ts）
      → UI（sidebar, worktree 面板, 操作页面切换）
```

但依赖链中存在可并行段：
- 后端 git/fs 模块 ←→ 前端 sidebar/worktree UI 骨架 **可并行**
- 前端 IPC + stores ← 依赖后端命令签名稳定后
- 整体写入文件无冲突（3 类文件互不相交）

因此采用 **`/goal` 为主干锁 DoD + 1 个 dynamic workflow 处理全量实现**（3 agent 并行：后端 Rust / 前端基础设施 / 前端 UI），与 Phase 1 第三轮收口（18 号计划 §4）和 Phase 0 补救（§8）模式一致。

### 1.2 三层结构

```
外层  /goal .............. 锁 Phase 2 DoD，每轮自动判定，未达成则续轮，达成自动 clear
  中层  dynamic workflow .. 一次性后台编排：
        Phase 'Backend'  → Agent B: git 模块（git2 + shell worktree）+ fs 模块（原子持久化）
        Phase 'Frontend' → parallel(Agent F1: stores+ipc+types, Agent F2: sidebar/worktree UI)
        Phase 'Tests'    → parallel(Agent T1: Rust tests, Agent T2: TS tests)
        Phase 'Verify'   → parallel(cargo test, npm test, tsc, eslint, clippy, build)
```

### 1.3 版本前提与降级

| 能力 | 最低版本 | 自检 |
|------|---------|------|
| `/goal` | Claude Code 2.1.139+ | `claude --version` |
| dynamic workflow | 2.1.154+（Pro 在 `/config` 开） | `claude --version` + `/config` |

**降级**：若 `/goal` 不可用 → 改为人工逐项验收顺序推进（§4 各 agent 验收仍成立）；若 workflow 不可用 → 主会话顺序 spawn 3 agent（后端→前端→测试）。两者缺失不阻塞 Phase 2 主体。

## 2. 如何使用 `/goal`（主干）

### 2.1 机制要点

- `/goal` 是**会话级 Stop hook**：每轮结束，把「完成条件 + 对话 transcript」发给 Haiku，返回 yes/no + reason；未达成自动开下一轮。
- **Evaluator 只看对话中已暴露的命令输出**——不自己跑命令、不读文件。
- ⇒ **凡声称「通过」的项，必须把真实命令输出贴回主会话**。这条贯穿 §4 所有 agent。
- 条件只放自动化命令门（exit 0 + 文本匹配），grep/源码检查移人工验收（plan-generation.md §6.19）。

### 2.2 启动指令

以下为可直接复制到 Claude Code 中执行的完整 `/goal` 指令：

```
/goal 按 plan/execute/Phase2/1.执行计划.md §4 变更清单
完成 Phase 2「侧边栏与项目结构」——后端 git 模块（git2 读 + shell worktree 写）
+ fs 原子持久化 + 前端 sidebar/worktree UI + projects/layout stores。

until:
  cargo test -- --test-threads=1 exits 0，输出含 "test result: ok" 且 "0 failed"
  npm test exits 0，输出含 "0 failing"
  npm run test:l3 exits 0，输出含 "0 failing"
  npx tsc --noEmit exits 0
  npx eslint src/ exits 0
  cargo clippy -- -D warnings exits 0
  cargo build --debug exits 0
  npm run wdio exits 0（spec 必须 PASS；teardown 噪声可搁置——
    已知 @wdio/tauri-service session 销毁时序问题，不影响 spec）

hard constraints:
  - 不得弱化任何现有测试断言，test 数量不得减少（Phase 1 回归不退化）
  - 不得放宽任何 lint 规则（eslint.config.js / Cargo.toml lint 部分不得修改）
  - 不得修改 src-tauri/src/pty/ / src-tauri/src/claude/ / plan/ 目录下任何文件
  - 不得修改 src/__tests__/ 中 Phase 1 已有测试的断言逻辑（可新增 Phase 2 测试文件）
  - 不对 npm/cargo 依赖做非 Phase 2 必需的升级
  - W0 基线检查先 cargo test && npm test：若基线不绿，先修基线直到全绿再进 W1
  - W1 实现必须使用 Workflow 工具 3 agent 并行——
    后端 git/fs（Rust）、前端基础设施（stores+ipc+types）、前端 UI（sidebar+worktree）
    零文件冲突，串行手动改浪费时间
  - W2 回归阶段 workflow 子 agent 必须把 cargo test/npm test/test:l3/tsc/eslint/
    clippy/build 完整命令输出贴回主会话——
    Evaluator 看不见子 agent 内部对话，不回灌则判定"无验证输出"

or stop after 30 turns 并逐项汇报缺口。
```

### 2.3 执行流程

`/goal` 是单条命令——Evaluator 只看终态条件是否满足。但 Agent 在自主循环内按以下节律工作：

```
Turn 1+: W0 基线 → cargo test + npm test（贴输出）→ 基线全绿
         若基线不绿 → 先修基线直到全绿再进 W1

Turn 2+: W1 实现 → 调用 Workflow 工具 3 agent 并行
         Agent B: 后端 git 模块 + fs 原子持久化（Rust）
         Agent F1: 前端 stores + ipc + types（基础设施层）
         Agent F2: 前端 sidebar + worktree UI + Workspace 集成

         W2 回归 → workflow 回归阶段并行验证
         cargo test ∥ npm test ∥ test:l3 ∥ tsc ∥ eslint ∥ clippy ∥ build
         → 子 agent 必须把全部命令输出贴回主会话

Turn 3+: W3 失败修复 → 若 Evaluator 判定条件未全满足
         → Agent 修复 → 贴修复后命令输出 → 下一轮评估

Turn N:  W4 完成 → Evaluator 判定全部条件满足
         或 30 turns 后停止并逐项汇报缺口
```

## 3. 如何用 dynamic workflow（旁路）

### 3.1 使用边界

- 触发：prompt 含 Workflow 工具调用（Agent 在 W1 自动触发）
- 并发 ≤ 16、单次 ≤ 1000 agent
- 子 agent **强制 `acceptEdits`**，继承主会话 tool allowlist
- 跨会话不续 → **单会话内跑完**

## 4. 各 agent 变更清单

### 4.1 Agent B — 后端（Rust）

#### 4.1.1 git/mod.rs —— git2 读操作

**操作**：
1. 删除 `// 模块占位` 注释
2. 实现 `git_is_repo(path: String) -> Result<bool, AppError>`：`Repository::open(path).is_ok()`
3. 实现 `git_root(path: String) -> Result<String, AppError>`：`repo.workdir().map(|p| p.to_string_lossy().to_string())`
4. `pub mod worktree;` 声明子模块

**验收**：`cargo build --debug` 通过，`cargo test` 含 git_root 单元测试 PASS

#### 4.1.2 git/worktree.rs（新建）—— shell worktree 写操作

**操作**：
1. 实现 `git_worktree_list(repo_path: String) -> Result<Vec<WorktreeInfo>, AppError>`
   - 解析 `git worktree list --porcelain` 输出
   - 路径反斜杠归一化为正斜杠
   - `WorktreeInfo { path, branch, head, is_bare, is_detached, is_main }`
2. 实现 `git_worktree_add(repo_path: String, name: String) -> Result<WorktreeInfo, AppError>`
   - 净化分支名：`[^A-Za-z0-9/._-]` → `-`
   - 路径：`.claude/worktrees/<name>`，分支：`worktree-<name>`
   - 30s 超时（`wait-timeout` crate）
   - `git worktree add <path> <branch>`（不用 `-b`——分支应不存在）
3. 实现 `git_worktree_remove(repo_path: String, name: String) -> Result<(), AppError>`
   - `git worktree remove --force <path>` + `git worktree prune`
   - 若删除失败：`attrib -R <path>\* /S /D` → `cmd.exe /c rmdir /S /Q <path>`
   - **绝对禁止** `Remove-Item -Recurse -Force`（穿透 NTFS junction）

**DTO**：
```rust
#[derive(Debug, Serialize, Deserialize)]
#[serde(rename_all = "camelCase")]
pub struct WorktreeInfo {
    pub path: String,
    pub branch: String,
    pub head: String,
    pub is_bare: bool,
    pub is_detached: bool,
    pub is_main: bool,
}
```

**验收**：`cargo build --debug` 通过，集成测试 PASS

#### 4.1.3 fs/mod.rs —— 原子持久化

**操作**：
1. 添加 `use tempfile::NamedTempFile; use std::io::Write;`
2. 实现 `save_settings(settings: serde_json::Value) -> Result<(), AppError>`
   - 获取 `app_data_dir` → `settings.json`
   - 写临时文件 → flush → persist（覆盖原文件）
   - persist 前 copy 原文件为 `.bak`
3. 实现 `load_settings() -> Result<serde_json::Value, AppError>`
   - 读取 `settings.json` → JSON parse
   - 若 parse 失败 → 尝试从 `.bak` 恢复 → 仍失败则返回 `Null`
4. 将现有 `fs_write_file` 改为原子写入（复用 tempfile 模式）

**验收**：`cargo test` 含 fs_read_write_roundtrip PASS，`cargo build --debug` 通过

**操作**：

## 7. 注意事项与已知坑

### 7.1 Windows worktree 安全清单

| 优先级 | 坑 | 规避 |
|--------|---|------|
| 🔴 CRITICAL | **NTFS junction 灾难**：`Remove-Item -Recurse -Force` 和 `rm -rf` 会穿透 pnpm `node_modules` 中的 NTFS junction，删除用户文件（Claude Code #29249 曾删光用户 home 目录） | **仅用** `cmd.exe /c "rmdir /S /Q <path>"` 或 `git worktree remove` |
| 🔴 CRITICAL | **分支名非法字符**：`~ ^ : ? * [ \` 空格等 | 净化：`[^A-Za-z0-9/._-]` → `-`；`git check-ref-format --branch` 预校验 |
| 🔴 CRITICAL | **MCP/子进程 CWD 锁定**：子进程 CWD 在 worktree 内时 `git worktree remove` 返回 Permission denied（Claude Code #32747） | 关闭顺序：先 kill 子进程 → wait 1-2s → 再 remove |
| 🟡 HIGH | **`tempfile::persist()` 非真原子**：Windows `MoveFileExW` 目标存在时失败，tempfile 退化为先删后 rename（中间有窗口期目标文件不存在） | `.bak` 备份兜底：persist 前 copy 原文件为 `.bak`，启动时若主文件损坏从 `.bak` 恢复 |
| 🟡 HIGH | **worktree 目录的 `.git` 文件约定**：必须用**正斜杠** `gitdir: C:/path/to/.git/worktrees/id`、必须用**LF 换行**（CRLF 导致 IDE git 集成完全失效） | 创建时确保 UTF-8 LF 输出 |
| 🟡 HIGH | **`git worktree list --porcelain` 路径反斜杠**：Windows 上 worktree 行输出反斜杠路径 | 解析后归一化为正斜杠 |
| 🟡 HIGH | **`gitdir:` 使用绝对路径**：不支持相对路径——`git worktree list` 输出是绝对路径 | 移动仓库目录后需 `git worktree repair` 修复 gitdir 引用 |
| 🟢 MEDIUM | **fsmonitor--daemon 文件锁定**：Git ≥ 2.38 默认启用，持有 `.git/fsmonitor--daemon.ipc` 句柄 | remove 前执行 `git fsmonitor--daemon stop` |
| 🟢 MEDIUM | **杀毒软件/搜索索引器锁定**：Defender/Indexer 在文件创建后短时间内持有句柄 | 添加排除项；重试 3 次间隔 1-3s 指数退避 |
| 🟢 LOW | **`core.longpaths` 未启用**：路径 > 260 字符时工作树创建失败 | `git config --global core.longpaths true` |

### 7.2 终端保活三层防御

| 场景 | 方案 | 机制 |
|------|------|------|
| 同组页签切换 | `renderer: 'always'` | xterm.js DOM 保持挂载，`display:none`/`display:block` 切换，PTY 零中断 |
| 操作页面切换（`fromJSON`） | `reuseExistingPanels: true` | 面板 ID 匹配时保留实例，Dockview 不销毁 Panel 组件 |
| 强制重建（面板 ID 不匹配） | SerializeAddon | 序列化终端画面 → 新 Terminal 实例 → `write(serialized)` → 新 PTY spawn（cwd 从 binding 恢复） |

### 7.3 数据流边界

```
Rust 后端          IPC（invoke）       前端 Zustand store      React UI
─────────         ──────────         ──────────────         ────────
git_is_repo ──→   ipc/git.ts  ──→   projects store  ──→   SidebarTree
git_worktree_list→ ipc/git.ts  ──→   projects store  ──→   SidebarTree + WorktreeActions
git_worktree_add ─→ ipc/git.ts  ──→   projects store  ──→   SidebarTree（re-render）
git_worktree_remove→ipc/git.ts ──→   projects store  ──→   SidebarTree（remove node）
save_settings ←──  ipc/settings.ts ←─ projects store persist ←─ 自动 debounce
load_settings ──→  ipc/settings.ts ──→ projects store init ←─ 启动时
```

### 7.4 布局切换事件序列（~200-500ms）

```
用户点击操作页面
  → 0. saveLayout(当前) → 更新当前页面的 layout JSON
  → 1. setGlobalFlag('layoutSwitching', true) → 暂停 ResizeObserver
  → 2. api.clear() → 关闭所有面板
  → 3. api.fromJSON(目标布局, { reuseExistingPanels: true }) → 恢复面板
  → 4. 各 TerminalPanel 挂载 → useXterm 读取 params.binding.worktreePath
  → 5. pty.spawn({ cwd: worktreePath, ... }) → 终端在正确目录启动
  → 6. setGlobalFlag('layoutSwitching', false) → ResizeObserver 恢复
  → 7. 首个终端面板 focus()
```

### 7.5 删除 worktree 完整流程

```
用户点击"删除 worktree"
  → Phase 0: 预检——主 worktree？当前活跃 worktree？→ BLOCK
  → Phase 1: CAS 锁 acquireLock(worktreePath)
  → Phase 2: 前端确认对话框——列出受影响面板 + 未提交改动
  → Phase 3: Checkbox gate → 解锁[删除]按钮
  → Phase 4: 用户确认 → kill 所有关联 PTY → close() 所有关联 panels
  → Phase 5: git worktree remove --force <path>（最多重试 3 次）
  → Phase 6: 若目录仍存在 → attrib -R + cmd.exe /c rmdir /S /Q（最多重试 5 次）
  → Phase 7: git worktree prune
  → Phase 8: 清理 store（removeWorktree + removePage + removeSession）
  → Phase 9: 若活跃页面被删 → 切换到主工作区页面
```
