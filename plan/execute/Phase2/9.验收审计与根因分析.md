## 1. H6：终端会话在页面切换后不保留

### 1.1 问题描述

在一个操作页面的终端中操作后，切换到其他操作页面再切回，终端布局恢复但终端是全新启动的，没有继续之前的终端调用。

### 1.2 完整调用链（`switchToPage` 执行流程）

```
1. user 点击侧栏 Page B → SidebarTree.tsx:426 → switchToPage(projId, pageId)
2. 自切换守卫: activePageId === pageId → return (Workspace.tsx:198)
3. setLayoutSwitching(true) (Workspace.tsx:200)
4. flushDirtyLayout() → saveLayout → updatePageLayout → store (Workspace.tsx:204)
5. loadLayout(api, pageB.layout) → fromJSON(layoutB, {reuseExistingPanels: true})
   └─ Dockview 源码: 先收集 ID 匹配面板到临时组, 再 clear() 销毁其余, 最后反序列化
      (dockviewComponent.js:1164-1186)
6. Page A 面板因 ID 不匹配被全部销毁 → React unmount → useXterm cleanup
7. setLayoutSwitching(false) (Workspace.tsx:223)
8. 新 TerminalPanel 挂载 → useXterm → 创建新 Terminal → pty.spawn() 新 session
```

### 1.3 根因树

#### ✅ 已确认：5 层连锁根因

**第 1 层——Panel ID 不稳定（`Date.now()` 时间戳）**

| 位置 | ID 生成 | 稳定性 |
|------|---------|--------|
| `Workspace.tsx:67`（Watermark 新建终端） | `terminal-${Date.now()}` | ❌ 每次不同 |
| `Workspace.tsx:90`（Watermark 新建编辑器） | `editor-${Date.now()}` | ❌ 每次不同 |
| `Workspace.tsx:119`（工具栏 + 按钮） | `terminal-${Date.now()}` | ❌ 每次不同 |
| `Workspace.tsx:257`（onReady 默认终端） | `terminal-init-${Date.now()}` | ❌ 每次不同 |
| `SidebarTree.tsx:308`（默认布局） | `terminal-${pageId}-0` | ✅ 跨切换稳定 |

**后果**：`reuseExistingPanels: true` 只通过 panel ID 字符串匹配保护面板（不检查组件类型、params）。除默认布局的第一个终端外，所有用户创建的终端 ID 跨页面永不匹配 → Dockview 必然销毁它们。

来源：Dockview 源码 `dockviewComponent.js:1164-1186` 确认——`fromJSON` 内部先收集 ID 匹配的面板到临时组保护，再 `clear()` 销毁其余，最后反序列化。

**第 2 层——`term.dispose()` 无 `isLayoutSwitching` 保护（`useXterm.ts:285-312`）**

```typescript
// useXterm.ts:285-312 — cleanup 函数
return () => {
  // ... 清理 raf/timer ...
  resizeObserver.disconnect();
  if (sessionIdRef.current) {
    const isSwitching = useLayout.getState().isLayoutSwitching;
    if (!isSwitching) {
      pty.kill(sessionIdRef.current);  // ✅ 有保护 — 页面切换时 PTY 进程存活
    }
  }
  setActiveTerminal(null);
  webglAddonRef.current?.dispose();     // ❌ 无保护
  fitAddonRef.current?.dispose();       // ❌ 无保护
  term.dispose();                       // ❌ 无保护——xterm.js dispose() 永久不可逆
  terminalRef.current = null;
};
```

xterm.js `dispose()` 是单向"陷阱门"（trapdoor）。销毁后 `write()` 被覆写为 no-op、WebGL 上下文丢失、DOM 元素移除、所有 addon 递归 dispose。**不存在 `reopen()` 或 `reattach()` API**（xterm.js Discussion #4739 官方确认）。

**第 3 层——Tauri Channel 无法重连（`reader.rs:40-43` + `pty.ts:16-19`）**

```rust
// reader.rs:40-43 — Channel send 失败时 reader 线程退出
if on_output.send(PtyEvent::Output { bytes }).is_err() {
    break;  // Channel 已关闭（前端断开），停止读取 → 线程退出
}
```

- Channel 绑定单次 `invoke` 调用，API 层面不提供 `Clone` 或重连机制
- 前端组件卸载 → JS Channel 被 GC → Rust `send()` 返回 `Err` → reader 线程退出
- 即使 PTY 进程被 `isLayoutSwitching` 保护存活，输出无法送达前端（reader 线程已死）

**第 4 层——`TerminalRegistry` 模块不存在（`useXterm.ts:299,304`）**

```typescript
// useXterm.ts:299,304 — 孤儿注释引用不存在的模块
// N2: 布局切换时不杀 PTY（进程存活在 TerminalRegistry 中）
// switching 时：PTY 进程存活，由 TerminalRegistry.get(panelId) 查询复用
```

- `TerminalRegistry.ts` 已在 worktree 删除时被误删（见 `7.删除worktree功能开发计划.md` D9）
- `useXterm.ts:198-209` 每次挂载**无条件调用 `pty.spawn()`**——无已有 session 查询逻辑
- 无跨组件 session 跟踪——所有 session 引用仅存在于 React ref 中

**第 5 层——架构假设落空**

原架构假设：Dockview `reuseExistingPanels` 会保留面板状态 → 面板因为 ID 不匹配被销毁 → React 组件卸载 → cleanup 无条件 dispose xterm → Channel 断开导致 reader 线程退出 → PTY session 变成"僵尸"（进程存活但无输出管道）。

#### ❌ 已排除的假设

| 假设 | 排除原因 |
|------|---------|
| `reuseExistingPanels` 不工作 | 机制本身正确——完全匹配 ID 的面板确实被保留。问题是 ID 不匹配。 |
| `handleRenamePage` 绕过 store 的 `renamePage` action | `SidebarTree.tsx:335` 正确调用 `renamePage()`——destructure 自 `useProjects((s) => s.renamePage)`（第 252 行） |

### 1.4 修复方案

**核心原则**：借鉴 Hyper、Tabby、VS Code——**标签切换时从不 dispose xterm.js 实例**。终端进程生命周期必须独立于 Dockview 面板生命周期。

**连 Tabby（66k stars）也不支持 live PTY reattach**——关闭标签 = 杀进程，恢复仅配置快照。Phase 2 优先走"可见性切换"（PTY + Terminal 常驻），Phase 3+ 实现 `pty_reattach`。

**推荐：进程与 UI 分离（VS Code 模式）**

```
┌─ Zustand Store ───────────────────────────────────┐
│  terminalRegistry: Map<panelId, {                  │
│    term: Terminal,          // xterm.js 实例       │
│    sessionId: string,       // PTY session ID      │
│    cwd: string,             // 工作目录            │
│  }>                                                │
└────────────────────────────────────────────────────┘
         │                    │
    mount 时取出          unmount 时存回
    xterm.open(el)        不调 dispose()
         │                    │
    ┌────▼────────────────────▼───┐
    │  TerminalPanel (React)      │
    │  - 挂载: registry.get(id)   │
    │  - 卸载: registry.set(id)   │
    │  - 仅真正关闭时才 dispose   │
    └─────────────────────────────┘
```

**分步实施**：

| # | 步骤 | 文件 | 复杂度 | 说明 |
|---|------|------|--------|------|
| A1 | 创建 `TerminalRegistry` 模块 | `src/panels/terminal/TerminalRegistry.ts`（新） | 中 | 模块级 `Map<panelId, { term, sessionId, webglAddon, fitAddon }>`。暴露 `register/get/remove/has` 方法。恢复 worktree 删除时被误删的模块 |
| A2 | `useXterm` 查 Registry 再决定 spawn | `useXterm.ts:198-209` | 中 | mount 时先查 Registry——若有已有 session 则复用 Term 实例（`term.open(el)`），若无才 `pty.spawn()` |
| A3 | `useXterm` cleanup 中移除 `term.dispose()` | `useXterm.ts:307-309` | 低 | 仅真正关闭（右键→关闭面板 / `pty_kill`）时才 dispose。`isLayoutSwitching` 时完全跳过 dispose |
| A4 | 稳定 Panel ID | `Workspace.tsx:67,90,119,257` + `SidebarTree.tsx:308` | 低 | `Date.now()` → `terminal-${pageId}-${seq}`（确定性）。全局维护 per-page 自增计数器 |
| A5 | `renderer: 'always'` 全局启用 | `Workspace.tsx` + `panelRegistry.ts` | 低 | Dockview 的 `renderer: 'always'` 使用 `visibility: hidden` 而非 `display: none`，保留 WebGL 上下文 |

**Phase 3+ 增强**（非 Phase 2 范围）：

| # | 功能 | 说明 |
|---|------|------|
| P3-1 | `pty_reattach` 命令 | Rust 后端支持替换 reader 线程的 Channel。需要 `Arc<RwLock<Option<Channel>>>` 包装 |
| P3-2 | Ring buffer 回放 | 后端缓存最近 N 行输出（如 1000 行），重连时回放 |
| P3-3 | 30s Canvas 延迟释放 | Tabby `visibility.ts` 模式：不可见时先保留 WebGL，超时后释放 |

**排除的路径**：

| 方案 | 排除原因 |
|------|---------|
| `fromJSON` monkey-patch 阻止 `clear()` | `clear()` 重置 gridview 内部状态（root BranchNode），反序列化依赖此干净状态。跳过会导致 "Invalid grid element" |
| 仅靠 `reuseExistingPanels` + 稳定 ID | Dockview 无 `onWillDispose`/`canDispose` 钩子。即使 ID 匹配，`updateFromStateModel()` 合并 params 行为不可预测 |
| SerializeAddon 序列化回放 | 不序列化光标可见性/OSC 颜色/窗口标题/终端尺寸。10k 行序列化需 ~50-170ms——不可每次切换执行 |
| 仅用 Event 替代 Channel | Tauri Event 全局广播无 per-panel 隔离。需自建 session_id 路由层，复杂度等同重写 Channel |

## 2. S1：删除操作页面后面板残留

### 2.1 问题描述

右键操作页面 → "删除操作页面"，页面从侧栏移除但操作面板仍显示被删除页面的内容。

### 2.2 根因

#### ✅ 已确认

**`handleDeletePage` 只操作数据模型，不操作 UI（`SidebarTree.tsx:323-328`）**

```typescript
const handleDeletePage = useCallback(
  (projectId: string, pageId: string) => {
    removePage(projectId, pageId);  // 仅删 store 中的数据
  },
  [removePage],
);
```

缺失三个必要步骤：

1. **未清空 Dockview 面板**：`api.clear()` 从未被调用。Dockview 完全不知道页面已被删除，继续显示被删页面的面板。

2. **未更新 `useLayout.activePageId`**：`removePage`（`projects.ts:119-142`）更新的是 `project.activePageId`（项目内部），但 `useLayout` store 的 `activePageId` 仍指向已删除的页面。`dirtyPageIdRef`（Workspace.tsx:280）引用了已删除的页面 ID——后续 `flushDirtyLayout` 搜索该 ID 时找不到对应页面，布局写入静默失败。

3. **未触发页面切换**：虽然 `removePage` 设置 `project.activePageId` 为下一个可用页面（`projects.ts:126`），但 `switchToPage` 从未被调用——Dockview 不知道要加载新页面的布局。

**SidebarTree 缺少 Dockview API 访问**

`SidebarTree` 只接收 `switchToPage` 作为 prop（`Workspace.tsx:328`），没有 `apiRef` 或 `api` 的引用。即使 `handleDeletePage` 想要调用 `api.clear()`，也没有途径。

#### ❌ 已排除

| 假设 | 排除原因 |
|------|---------|
| `removePage` store action 有 bug | Action 正确更新 `projects[projectId].pages`、`activePageId`、`expandedNodes` |

### 2.3 修复方案

**推荐：重组调用链——Workspace 层编排删除流程**

`SidebarTree` 不直接调 `removePage`，改为调 `Workspace` 传入的 `onDeletePage` 回调。Workspace 层持有 `apiRef`，可编排完整流程：

```typescript
// Workspace.tsx — 新增 onDeletePage 回调
const onDeletePage = useCallback((projectId: string, pageId: string) => {
  const api = apiRef.current;
  if (!api) return;

  const layoutStore = useLayout.getState();
  const projectsStore = useProjects.getState();
  const project = projectsStore.projects[projectId];
  if (!project) return;

  // 1. 若删除的是当前活跃页面 → 先清空 Dockview
  if (layoutStore.activePageId === pageId) {
    api.clear();
    layoutStore.setActivePage(null);
  }

  // 2. 从 store 移除
  projectsStore.removePage(projectId, pageId);

  // 3. 自动切换到剩余页面（若有）
  const updatedProject = useProjects.getState().projects[projectId];
  if (updatedProject?.activePageId) {
    switchToPage(projectId, updatedProject.activePageId);
  }
}, [switchToPage]);
```

然后 `SidebarTree.tsx` 中 `handleDeletePage` 改为调用 `onDeletePage` prop。

| # | 步骤 | 文件 | 复杂度 |
|---|------|------|--------|
| B1 | `Workspace` 新增 `onDeletePage` 回调 | `Workspace.tsx` | 低 |
| B2 | `SidebarTree` 接收 `onDeletePage` prop 替代直接调 `removePage` | `SidebarTree.tsx:323-328,245,307` | 低 |
| B3 | `SidebarTree` 类型定义增加 `onDeletePage` prop | `SidebarTree.tsx` | 低 |

## 3. S2：重命名操作页面卡死

### 3.1 问题描述

右键操作页面 → "重命名操作页面" → 输入新名称后，应用程序卡死，一段时间后恢复但名称未变更。

### 3.2 根因

#### ✅ 已确认

**`window.prompt()` 在 Tauri v2 中不可靠（`SidebarTree.tsx:332-338`）**

```typescript
const handleRenamePage = useCallback(
  (projectId: string, pageId: string) => {
    const newName = prompt("新页面名称:");   // ← 同步阻塞 API
    if (!newName?.trim()) return;
    renamePage(projectId, pageId, newName.trim());
  },
  [renamePage],
);
```

三重机制导致卡死：

1. **同步阻塞**：`prompt()` 通过同步 Mojo IPC 完全暂停 JS 主线程。所有 React 渲染、IPC 消息、事件处理在此期间冻结。

2. **Tauri 占位实现**：Tauri (wry) 的 `prompt()` 是**占位符**。macOS (WKWebView) 静默返回 null；Windows (WebView2) 行为不一致。Tauri 官方不将同步阻塞对话框作为一等 API——`plugins-workspace #2145` 明确标记为未实现。

3. **WebView2 兼容性**：WebView2 默认 JS 对话框与标准 Chromium 使用相同代码路径，但 `AreDefaultScriptDialogsEnabled=false`（Tauri 自定义模式）时通过 `GetDeferral()` 延长阻塞。若宿主在 UI 线程同步 `ShowDialog()`，触发重入死锁。

**`renamePage` store action 本身正确**（`projects.ts:165-181`）——问题 100% 在于 `prompt()` 的输入获取环节，不在于 store 更新。

#### ❌ 已排除

| 假设 | 排除原因 |
|------|---------|
| `renamePage` 不存在或绕过 | `SidebarTree.tsx:252` 正确 destructure，`projects.ts:165` 实现正确 |
| `tauri-plugin-prevent-default` 影响 prompt | 该插件仅处理键盘/鼠标快捷键（F3/Ctrl+J/右键），无 PROMPT flag |

### 3.3 修复方案

**推荐：内联编辑模式（最小改动，最佳 UX）**

`window.prompt()` 在 Tauri v2 全平台不可靠——macOS 静默返回 null，Windows 行为不一致。完全禁止使用。

**实现**：`PageRow` 组件中页面名称从 `<span>` 切换为 `<input>`：

```typescript
// SidebarTree.tsx — PageRow 内联编辑
const [editing, setEditing] = useState(false);
const [editValue, setEditValue] = useState("");

const startRename = () => {
  setEditValue(page.name);
  setEditing(true);
};

const confirmRename = () => {
  if (editValue.trim() && editValue.trim() !== page.name) {
    renamePage(projId, page.pageId, editValue.trim());
  }
  setEditing(false);
};

// JSX:
{editing ? (
  <input
    value={editValue}
    onChange={(e) => setEditValue(e.target.value)}
    onBlur={confirmRename}
    onKeyDown={(e) => {
      if (e.key === "Enter") confirmRename();
      if (e.key === "Escape") setEditing(false);
    }}
    autoFocus
    style={{ /* JetBrains 暗色配色 */ }}
  />
) : (
  <span>{page.name}</span>
)}
```

| # | 步骤 | 文件 | 复杂度 |
|---|------|------|--------|
| C1 | `PageRow` 增加 `editing`/`editValue` state + `<input>` 渲染 | `SidebarTree.tsx` | 低 |
| C2 | 右键菜单"重命名" → 调 `startRename()` 替代 `handleRenamePage` | `SidebarTree.tsx:435-437` | 低 |
| C3 | 删除 `handleRenamePage`（含 `prompt()` 调用） | `SidebarTree.tsx:331-338` | 低 |

**备选**：自定义 `usePrompt()` hook（Promise + createPortal + 暗色模态），适合多处需要文本输入的场景。引用 `theme/colors.ts` 配色。

## 4. S4：重启后操作页面为空白态

### 4.1 问题描述

重启后恢复到上次活跃的页面，但操作页面为空白态，没有保留关闭前的分屏布局。

### 4.2 根因树

#### ✅ 已确认：6 层连环根因

**第 1 层——`onDidLayoutChange` 不写 store（`Workspace.tsx:276-281`）**

```typescript
api.onDidLayoutChange(() => {
  if (restoreGuardRef.current) return;
  const activeId = useLayout.getState().activePageId;
  if (!activeId) return;
  dirtyPageIdRef.current = activeId;  // ← 仅设 ref，不调 updatePageLayout
});
```

用户拖拽分屏、添加面板、调整尺寸 → `onDidLayoutChange` 触发 → **仅设置 volatile ref**。Zustand store 从未收到布局变更 → 2s debounce subscribe 永不被触发 → 磁盘永不保存。

**第 2 层——`flushDirtyLayout()` 仅在 `switchToPage` 调用（`Workspace.tsx:204`）**

```typescript
// flushDirtyLayout 定义 (Workspace.tsx:171-188) —— 唯一的 layout→store 写入路径
// 调用点：仅 switchToPage:204 一处
```

如果用户始终在同一页面操作（不切换），关闭应用前布局**从未被持久化**。

**第 3 层——2s debounce subscribe 对布局变更无效（`projects.ts:272-278`）**

```typescript
useProjects.subscribe(() => {
  if (!initialized) return;
  if (saveTimer !== null) clearTimeout(saveTimer);
  saveTimer = setTimeout(() => { saveAllProjects(); }, 2000);
});
```

Zustand `subscribe` 只对 `set()` 调用响应。由于第 1 层不调用 `set()`，此订阅对布局变更**零作用**——它是被动观察者，等不到任何变化。

**第 4 层——`beforeunload` 不保存布局（`App.tsx:54-63`）**

```typescript
const handleBeforeUnload = () => {
  const activePageId = useLayout.getState().activePageId;
  if (activePageId) {
    localStorage.setItem("slterm-last-active-page", activePageId);  // ← 仅此
  }
};
```

只保存 `activePageId`（字符串）。不调 `flushDirtyLayout`、不调 `saveAllProjects`、不写文件。即使调了 `flushDirtyLayout`，2s debounce 的 `setTimeout` 在 `beforeunload` 中也不会等到执行——窗口在 timeout 触发前就关闭了。

**第 5 层——启动竞态（`App.tsx:40-50` vs `Workspace.tsx:227-265`）**

```
timeline:
  1. React render → Workspace 挂载 → DockviewReact 挂载 → onReady 触发
  2. useEffect 开始: loadAllProjects() 异步读文件...（尚未完成）
  3. onReady 中: useProjects.getState().projects === {}（空）
  4. restored = false（找不到任何 project/page）
  5. isReadyRef.current = true（永久阻断后续恢复）
  6. ... 几毫秒后 ... loadAllProjects() 完成
  7. projects store 现在有数据了 — 但 onReady 已执行完毕
  8. 无任何 trigger 重新尝试恢复 → 用户看到空白 Watermark
```

**第 6 层——无 save-on-close 保证**

存在 4 个可能关闭时机，均不保证布局已保存：
- 用户正常关闭窗口 → `beforeunload`（不保存布局）
- 任务管理器杀进程 → 无任何钩子
- 系统关机 → 可能触发 `beforeunload`（不可靠）
- 应用崩溃 → 无任何钩子

2s debounce subscribe 是唯一防线——但它要求 store 被 mutate（第 1 层阻止了这点），且依赖进程在 debounce 窗口内未终止。

### 4.3 修复方案

**四层双保险持久化（借鉴 VS Code 多出口防线）**

```
┌─────────── Zustand Store ──────────┐
│  subscribe(listener)  getState()   │
└───────┬───────────────┬───────────┘
        │               │
  Layer 1              Layer 3          Layer 4
  subscribe+debounce   onCloseRequested  setInterval
  (2s debounce)        (flush dirty)    (30s)
        │               │                │
  Layer 2               │                │
  onDidLayoutChange →   │                │
  updatePageLayout()    │                │
        │               │                │
        └───────┬───────┴────────┬───────┘
                │                │
           invoke()  Tauri IPC
                │
         Rust write_json_atomic()
       (tempfile → sync → persist)
```

**必须修复（4 项）**：

| # | 步骤 | 文件 | 复杂度 | 说明 |
|---|------|------|--------|------|
| D1 | 启动竞态：`App` 加 `ready` state，条件渲染 `Workspace` | `App.tsx:40-50` | 中 | `loadAllProjects()` 完成前显示 `<LoadingScreen />`（全屏暗色，JetBrains 配色）。完成后 `setReady(true)` → 渲染 `Workspace` |
| D2 | `onReady` 移除 `isReadyRef` 防护 | `Workspace.tsx:227-265` | 低 | 渲染 `Workspace` 时数据已就绪，不再需要重试防护（`restoreGuardRef` 保留——fromJSON 恢复守卫仍需） |
| D3 | `onDidLayoutChange` → 调用 `updatePageLayout` | `Workspace.tsx:276-281` | 低 | 替换 `dirtyPageIdRef.current = activeId` → `updatePageLayout(projId, activeId, saveLayout(api))`。此改动触发 2s debounce→保存管道 |
| D4 | 关闭钩子：`onCloseRequested` → flush → 同步保存 | `App.tsx` + Rust `lib.rs` | 中 | 前端 `onCloseRequested(async (evt) => { evt.preventDefault(); await flushAndSave(); await appWindow.destroy(); })`。`destroy()` 而非 `close()` 防无限循环。Rust 双出口覆盖 `WindowEvent::CloseRequested` + `RunEvent::Exit` |

**建议增强（3 项）**：

| # | 步骤 | 文件 | 复杂度 | 说明 |
|---|------|------|--------|------|
| D5 | 原子文件写入 | `src-tauri/src/fs/atomic.rs`（新） | 中 | `tempfile::NamedTempFile::new_in(dir)` → `write_all` → `sync_all()` → `persist(path)`。同目录 rename 保证 NTFS `$LogFile` crash 安全 |
| D6 | 启动清理残留 `.tmp` 文件 | Rust `setup` / `run_on_event` | 低 | 扫描 app data dir，删除 `.tmp` 后缀文件 |

```
1. App mount → show <LoadingScreen />
2. useEffect: await loadAllProjects() (async file read)
3. markPersistenceReady()
4. setReady(true) → render <Workspace />
5. DockviewReact mount → onReady fires
6. useProjects.getState().projects 已有数据 ✅
7. activePageId 匹配 → loadLayout(api, page.layout) ✅
8. 分屏布局恢复成功 ✅
```

**`onCloseRequested` 实现参考**：

```typescript
// App.tsx
import { getCurrentWindow } from "@tauri-apps/api/window";

const appWindow = getCurrentWindow();
appWindow.onCloseRequested(async (event) => {
  event.preventDefault();
  // 1. flush dirty layout
  const { activePageId } = useLayout.getState();
  if (activePageId && window.__dockviewApi) {
    const layout = saveLayout(window.__dockviewApi);
    const { projects } = useProjects.getState();
    for (const [, proj] of Object.entries(projects)) {
      if (proj.pages.some((p) => p.pageId === activePageId)) {
        useProjects.getState().updatePageLayout(proj.projectId, activePageId, layout);
        break;
      }
    }
  }
  // 2. 同步保存到磁盘（绕过 debounce）
  saveAllProjects();
  // 3. 保存 activePageId
  localStorage.setItem("slterm-last-active-page", activePageId ?? "");
  // 4. 强制销毁（destroy 不会重新触发 onCloseRequested）
  await appWindow.destroy();
});
```

Rust 端双出口：

```rust
// lib.rs — 补充在 Builder 链中
.on_window_event(|window, event| {
    if let tauri::WindowEvent::CloseRequested { .. } = event {
        // 路径 A: 窗口 X 按钮 / Alt+F4
        flush_app_state(window.app_handle());
    }
})
.run(|app_handle, event| {
    match event {
        tauri::RunEvent::ExitRequested { .. }
        | tauri::RunEvent::Exit => {
            // 路径 B: macOS Cmd+Q / taskkill / 最后窗口关闭
            flush_app_state(app_handle);
        }
        _ => {}
    }
});
```

**Zustand `persist` 中间件评估**：完全兼容 Tauri——`StateStorage` 接口原生支持 async。但不推荐 Phase 2 引入——需重构持久化层、处理 hydration 闪白、`JSON.stringify` 每次 `set()` 执行。保持现有 `subscribe` + invoke 模式更可控。
