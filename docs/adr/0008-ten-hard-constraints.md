# 10 条硬约束作为可执行架构规则

Status: Accepted

项目的 10 条编码约束不是风格建议——它们是硬性规则，每一条对应一个可验证的检查点，由代码审查和自动化工具强制执行。

**为什么需要硬约束？** 项目由多 agent 并行开发，不同 agent 在不同会话中处理不同模块。没有机器可执行的架构约束，每一次修改都可能系统地侵蚀架构边界——比如 `invoke` 调用扩散到 IPC 层之外、平台条件编译泄漏到业务逻辑。硬约束将架构决策转化为可 grep/lint 的规则。

**备选方案**：仅靠代码审查（不可靠，审查者不知道所有边界规则）；ArchUnit 式架构测试（Rust/TypeScript 缺乏成熟的架构测试框架）。

**每条约束的检查方式**：
- #1（IPC 边界）：ESLint `no-restricted-imports` + grep `invoke` outside `src/ipc/`
- #2（模块隔离）：Rust `pub(crate)` 可见性 + 模块目录边界
- #5（面板封闭）：`PANEL_TYPES` 白名单校验 `fromJSON`
- #6（配色单点）：grep 色值硬编码
- #9（平台分支）：grep `#[cfg(windows)]` 仅出现在 `pty/`

**后果**：新增功能必须逐条对照 10 条约束。约束编号是位置敏感的（按根 CLAUDE.md 中顺序），不可随意重排。
